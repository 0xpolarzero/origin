# 03 Spec: Origin Namespace Isolation, Manual OpenCode Imports, and Branding Alignment

## Goal

Stop implicit settings/data sharing with OpenCode by moving Origin to its own namespace, while adding explicit one-click imports for only:

- providers/auth
- model visibility toggles

Also apply a targeted branding pass so Origin is branded as "origin", with explicit exceptions for upstream product references that must remain (for example, OpenCode Zen).

## Why This Phase Exists

The current fork still uses multiple OpenCode namespaces (`opencode` XDG roots, desktop store file names, app storage keys, desktop bundle identifiers). This creates accidental overlap with a user's real OpenCode installation.

We want explicit, user-controlled portability instead of implicit sharing:

- no shared runtime namespace by default
- selective import buttons for the two allowed domains
- clear product branding for the fork, without breaking upstream provider/service references

## In Scope

- Introduce Origin-owned runtime/settings namespaces for desktop app usage:
  - core global app directory namespace (`origin` instead of `opencode`)
  - desktop/local persisted store names and key prefixes (`origin.*` instead of `opencode.*`)
  - desktop bundle identifiers (`ai.origin.desktop*` instead of `ai.opencode.desktop*`)
- Keep Origin runtime isolated by default:
  - no automatic fallback to OpenCode settings
  - no background sync with OpenCode
  - no automatic migration of all OpenCode settings/state
- Add explicit settings actions:
  - Providers tab: `Load OpenCode providers`
  - Models tab: `Load OpenCode model toggles`
  - both import controls use compact button styling with an adjacent help/info tooltip clarifying OpenCode import context for this machine
- Provider import behavior:
  - import provider configuration from OpenCode using OpenCode config precedence semantics (project + `.opencode` + global, with OpenCode namespace paths)
  - import auth credentials from OpenCode auth store
  - auth merge is non-destructive: never overwrite existing Origin auth for the same provider
  - after a successful import, Origin must trigger the same global refresh/dispose path used by provider connect/disconnect so connected-provider UI state updates immediately without app restart
- Model import behavior:
  - import enabled/disabled model toggles from OpenCode model-toggle source in the OpenCode desktop app namespace/container
  - when multiple OpenCode desktop app identifiers are present on disk (`ai.opencode.desktop*` variants), merge valid `model.user` toggle entries across containers with deterministic precedence (preferred container first, then fallback containers) so dev/prod/beta toggles are all considered
  - apply imported toggles to Origin model-toggle store only
  - do not import unrelated model UX state unless explicitly specified
- Branding alignment pass for user-facing Origin product strings:
  - replace OpenCode-as-app naming with Origin-as-app naming where it refers to this forked app
  - preserve intentional upstream references (see keep set)
- Add tests and validation coverage for:
  - namespace isolation
  - provider/auth import merge semantics
  - model toggle import behavior
  - branding exceptions

## Out of Scope

- MCP import (config or auth) from OpenCode.
- Agent import from OpenCode (`agent` config, `.opencode/agent` content).
- Command/plugin/permission import from OpenCode.
- Session/message/history/snapshot/workspace DB migration from OpenCode into Origin.
- Full package ecosystem rename (`@opencode-ai/*`, `opencode` CLI binary name, full env var family rename).
- API/protocol/header family rename (`x-opencode-*`) in this phase.
- Replacing upstream product/service labels that are intentionally external references (for example OpenCode Zen).

## Baseline Assumptions

- This repository is desktop-first; import actions are designed for desktop runtime first.
- After namespace split, Origin starts from Origin-owned storage; OpenCode state is only used via explicit import actions.
- OpenCode provider/auth import source mirrors OpenCode behavior in OpenCode namespace paths:
  - project `opencode.json{,c}`
  - `.opencode/opencode.json{,c}` overlays
  - OpenCode global `~/.config/opencode/opencode.json{,c}`
  - OpenCode auth store `~/.local/share/opencode/auth.json` (platform equivalent)
- `Load OpenCode providers` merges auth entries without replacing existing Origin entries for the same provider ID.
- OpenCode model-toggle import source is OpenCode desktop global store `opencode.global.dat` key `model` read from the OpenCode desktop app identifier namespace (`ai.opencode.desktop*`), not Origin app storage namespace.
- If multiple OpenCode desktop app identifier containers exist (`ai.opencode.desktop`, `ai.opencode.desktop.dev`, `ai.opencode.desktop.beta`), model toggle import merges valid `user` entries across those containers with deterministic precedence (preferred container first).
- If the OpenCode model-toggle source is missing/invalid, model-toggle import is a no-op with explicit user feedback.
- `Load OpenCode model toggles` imports visibility toggles, not full model preferences.
- OpenCode references that indicate upstream provider/services (for example `OpenCode Zen`, provider IDs like `opencode` and `opencode-go`) must remain unchanged.
- Existing project-local compatibility for `.opencode` and `opencode.json{,c}` remains in place for this phase.

## Required Keep Set

- Keep upstream provider/service identifiers:
  - provider IDs `opencode` and `opencode-go`
  - user-facing `OpenCode Zen` references in model/provider UI where they refer to the upstream offering
- Keep compatibility readers for OpenCode config/auth/model sources only as explicit import sources, resolving from OpenCode namespace paths/containers.
- Keep `.opencode` and `opencode.json{,c}` compatibility behavior in core config loading for this phase.
- Keep current environment variable namespace (`OPENCODE_*`) and wire headers (`x-opencode-*`) in this phase.

## Deletion Candidates

Not applicable. This phase is additive/refactor-focused and does not require package/directory pruning.

## Agent Execution Plan

## Step 1: Establish Namespace Boundaries

- Introduce Origin runtime namespace constants and replace OpenCode namespace usage at:
  - core global path root
  - app persisted storage names/prefixes
  - desktop store names
  - desktop bundle identifiers
- Ensure new Origin runtime writes only to Origin-owned locations/keys.

Suggested checks:

```bash
rg -n "opencode\\.global\\.dat|opencode\\.settings\\.dat|const app = \"opencode\"|ai\\.opencode\\.desktop" packages/app packages/desktop packages/opencode/src
```

## Step 2: Add Explicit OpenCode Import Sources

- Add read-only import helpers for OpenCode provider/auth and model-toggle sources.
- Provider import source resolution must follow OpenCode precedence semantics under OpenCode namespace:
  - project `opencode.json{,c}`
  - `.opencode/opencode.json{,c}`
  - global OpenCode config files
  - OpenCode auth store
- Model toggle import source resolution must read OpenCode app-namespace storage (`ai.opencode.desktop*`), not Origin app storage namespace.
- Parse source payloads defensively; invalid entries are skipped, not fatal.
- Do not wire any automatic sync or implicit fallback behavior.

Suggested checks:

```bash
bun --cwd packages/opencode run test
```

## Step 3: Implement `Load OpenCode providers`

- Add button/action in providers settings.
- Keep the provider import control visually low-emphasis:
  - compact button size
  - adjacent help/info tooltip copy explaining OpenCode is the upstream coding app and imports apply when OpenCode was used on this machine
- Import provider config and auth from OpenCode sources resolved with OpenCode precedence semantics.
- Merge behavior for auth:
  - add missing provider auth entries
  - never overwrite existing Origin auth entries
- Return/import summary counts (imported, skipped, invalid).
- On success, force a live provider-state refresh (same event path as connect/disconnect) so newly imported auth/providers are visible immediately in settings without restarting the app.

Suggested checks:

```bash

```

## Acceptance Criteria

- `AC-01` Origin runtime namespace is separated from OpenCode namespace for core paths and app stores.
  - Verification:
    - automated tests assert new namespace constants/paths and storage key/file names.
  - Pass condition:
    - tests prove Origin runtime/settings write to Origin namespace locations/keys and no longer default to OpenCode namespace constants.

- `AC-02` Provider import is explicit and non-destructive.
  - Verification:
    - automated test pre-seeds Origin auth with one provider credential
    - automated test pre-seeds OpenCode project/global config sources and OpenCode auth with overlapping and non-overlapping providers
    - test executes `Load OpenCode providers` action/API
  - Pass condition:
    - overlapping provider auth in Origin remains unchanged
    - missing OpenCode providers from resolved OpenCode precedence sources are added to Origin
    - import summary reports imported/skipped counts
    - provider connection state is reflected in settings immediately after successful import (no app restart required).

- `AC-03` Model toggle import is explicit and scoped.
  - Verification:
    - automated test pre-seeds multiple OpenCode app-namespace containers and validates merged `model.user` toggle import across containers

    - test executes `Load OpenCode model toggles` action
  - Pass condition:
    - Origin model visibility reflects imported toggles read from merged OpenCode app namespace/container storage sources

## Risk Register

- Namespace split may appear as a settings reset for existing fork users because no automatic full migration is performed.
- Cross-platform path resolution for reading OpenCode import sources can diverge (Linux/macOS/Windows differences).
- Branding sweep can accidentally replace intentional upstream references if exception rules are not enforced centrally.
- Desktop identifier changes can impact updater behavior and existing install coexistence.
- Partial rename strategy (keeping protocol/env/package compatibility names) can leave mixed terminology if not documented clearly.
