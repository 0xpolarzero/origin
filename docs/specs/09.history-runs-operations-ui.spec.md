# 09 Spec: History, Runs, and Operations UX

## Goal

Implement the History surface (`Runs` + `Operations`) with deterministic cross-linking, provenance filtering, and debug visibility behavior aligned with runtime contracts.

## Why This Phase Exists

Users need observable execution and file-change history before outbound actions and automation triggers are expanded. This phase makes runtime behavior inspectable and reviewable.

## In Scope

- History view with tabs:
  - Runs
  - Operations
- Default Operations filter:
  - app-authored only by default
  - explicit toggle for user-authored entries
- Minimum cross-link contract:
  - Operation -> Open run
  - Run with changes -> Open operation
- Debug run visibility controls:
  - hidden by default
  - reveal via `Show Debug Sessions`
  - deep-link behavior from notification action
- Duplicate-event row rendering in Runs:
  - visible as non-error event row
  - excluded from run execution counters
  - opens event details (not run-session navigation when no run exists)
- Sorting/pagination contract:
  - stable descending sort by primary timestamp with deterministic secondary tiebreaker
  - cursor-based pagination with stable continuation across refresh

## Out of Scope

- Draft approval/sending UX details (Phase 10).
- Trigger scheduling UX (Phase 11).
- Reconciliation reminder delivery logic (Phase 12).

## Baseline Assumptions

- Runtime emits stable linkage IDs and statuses from earlier phases.
- Existing sidebar/session layout can host History navigation.
- Existing titlebar/history component can be extended.
- This phase does not require Drafts navigation links; Draft route/link integration is handled in Phase 10.
- Debug visibility precedence is deterministic: in-view toggle overrides for current view, persisted setting is default for new views.
- Run execution counters include only true run rows and exclude duplicate-event rows.

## Agent Execution Plan

### Step 1: Build History Data Queries

- Add read models for runs/operations with stable sorting and pagination.
- Include provenance and linkage metadata in query response shapes.

### Step 2: Implement Runs Tab

- Render execution rows and duplicate-event rows distinctly.
- Exclude duplicate-event rows from execution counters.
- Support debug filtering controls.

### Step 3: Implement Operations Tab

- Default app-authored filter.
- Optional include-user-edits toggle.
- Render source markers and links.

### Step 4: Implement Cross-Link Actions

- Add deterministic navigation actions:
  - Open run
  - Open operation
- Ensure missing-link states are handled gracefully.

### Step 5: Add Debug Deep-Link Behavior

- `Open debug session` actions auto-enable debug visibility and focus target entry.

### Step 6: Validation Coverage

- component tests
- data query tests
- e2e navigation and filter tests

## Risk Register

- High-cardinality history data can cause list performance regressions.
- Link consistency can regress if IDs are missing in some legacy rows.
- Filter defaults can confuse users if toggle state persistence is inconsistent.
- Debug visibility deep-linking can produce focus bugs on list refresh.
