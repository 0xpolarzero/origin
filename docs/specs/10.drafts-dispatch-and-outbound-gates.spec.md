# 10 Spec: Draft Dispatch and Outbound Safety Gates

## Goal

Implement the full Draft lifecycle, explicit approve/send model, dispatch revalidation, and strict runtime outbound gating so external side effects are always controlled and auditable.

## Why This Phase Exists

Outbound safety is core product trust. This phase enforces that all external writes pass through Draft controls with policy lineage and exactly-once semantics.

## In Scope

- Draft lifecycle UI/runtime:
  - `pending`, `blocked`, `approved`, `auto_approved`, `sent`, `rejected`, `failed`
- Pending/Processed tab semantics:
  - Pending: `pending`, `blocked`, `approved`, `auto_approved`
  - Processed: `sent`, `rejected`, `failed`
- Explicit user actions:
  - approve (ready-to-send)
  - send now
  - reject
  - edit
- Approval invalidation on material edit.
- Dispatch-time revalidation:
  - policy
  - destination/action constraints
  - integration enabled-state
  - auth health
- Strict runtime outbound gate:
  - managed integration endpoints accept writes only from dispatcher with valid `draft_id` + `integration_attempt_id`
  - managed endpoint inventory is explicitly declared (adapter-owned host/path/method registry)
- Non-origin outbound hard rejection (`workspace_policy_blocked`) with user-visible remediation outcome.
- Auto-approve behavior and policy-default-deny handling.
- Policy lineage fields persisted for policy/dispatch decisions.

## Out of Scope

- Cron/signal scheduling mechanics (Phase 11).
- Debug reminder flow (Phase 12).
- Advanced governance retention/tamper-evidence enhancements.

## Baseline Assumptions

- Integration adapters can expose a single dispatch entrypoint controlled by backend.
- Policy engine scaffolding exists or can be added in this phase.
- Draft rows and run linkage tables exist from earlier phases.
- Legacy drafts can be migrated safely to lifecycle/tab semantics in this phase.
- API authorization boundaries for `approve` vs `send now` are enforceable at runtime entrypoints.
- Dispatcher can atomically validate draft/run/workspace/integration bindings before write.

## Agent Execution Plan

### Step 1: Implement Draft Lifecycle Runtime

- Add deterministic transition handlers.
- Enforce tab mapping semantics and terminal/non-terminal states.

### Step 2: Implement Approve/Send Split

- Separate `approve` and `send now` actions in API/UI.
- Keep `approved`/`auto_approved` in Pending until terminal dispatch outcome.
- `send now` is allowed only from `approved` and `auto_approved`.

### Step 3: Implement Invalidation and Revalidation

- Material edit after approval resets to `pending`.
- Dispatch revalidates policy/auth/enabled-state/destination.
- Fail closed to `blocked` with reason code when revalidation fails.

### Step 4: Implement Strict Dispatcher Gate

- Reject direct outbound writes without valid draft context identifiers.
- Enforce Origin-only workspace outbound gate.
- Validate context binding on dispatch:
  - `draft_id` exists and is dispatch-eligible
  - `integration_attempt_id` belongs to same run/workspace/integration context
  - integration endpoint is in managed endpoint inventory

### Step 5: Implement Auto-Approve Rules

- Auto-approve default OFF.
- Apply adapter policy constraints.
- Policy-eval failure -> `blocked` default-deny.

### Step 6: Test Coverage

- lifecycle transitions
- edit invalidation
- revalidation failures
- non-origin hard-block
- outbound exactly-once behavior

## Acceptance Criteria

- `AC-01` Draft state machine is deterministic and rejects illegal transitions.
  - Verification: transition matrix unit tests.
  - Pass condition: all invalid transitions fail and valid transitions succeed.

- `AC-02` Approve and send are separate actions.
  - Verification: UI/service tests.
  - Pass condition: `approve` does not dispatch; `send now` dispatches only from `approved` and `auto_approved`.

- `AC-03` Material edit invalidates approval.
  - Verification: approve draft, edit material field, inspect state.
  - Pass condition: state resets to `pending` and requires re-approval.

- `AC-04` Dispatch revalidation fail-closes.
  - Verification: revoke auth/disable integration/change policy after approval.
  - Pass condition: dispatch becomes `blocked` with explicit reason and no external write.

- `AC-05` Direct outbound calls without draft context are rejected.
  - Verification: integration endpoint test without valid `draft_id` + `integration_attempt_id`.
  - Pass condition: call rejected and audit event emitted.

- `AC-06` Forged context mismatches are rejected.
  - Verification: dispatch attempts with valid-looking but mismatched IDs across workspace/integration/run.
  - Pass condition: request is rejected with deterministic policy error and no outbound write.

- `AC-07` Non-origin outbound attempts are rejected and visible.
  - Verification: attempt outbound action in standard workspace.
  - Pass condition: deterministic `workspace_policy_blocked` outcome with remediation hint.

- `AC-08` Policy lineage fields are persisted for allow/deny/dispatch decisions.
  - Verification: audit/event query tests.
  - Pass condition: `policy_id`, `policy_version`, `decision_id`, `decision_reason_code` are present and immutable for those events.

- `AC-09` Revalidation failures produce zero side effects.
  - Verification: induce revalidation failure after approval.
  - Pass condition: no external write occurs and deterministic blocked/failure code is recorded.

- `AC-10` Concurrent send requests are exactly-once.
  - Verification: concurrent `send now` requests and restart/retry race tests.
  - Pass condition: at most one outbound side effect occurs for one logical attempt.

- `AC-11` Managed endpoint inventory is fail-closed.
  - Verification: attempt outbound dispatch to endpoint not in adapter-managed inventory.
  - Pass condition: request is rejected with deterministic policy error and no outbound write occurs.

- `AC-12` Policy-evaluation failure at decision time is default-deny.
  - Verification: force policy evaluation failure before approval/auto-approval decision.
  - Pass condition: draft transitions to `blocked` and no dispatch is attempted.

- `AC-13` Validation commands pass.

  - Pass condition: all commands pass.

## Risk Register

- Dispatch gate bypasses can occur if legacy outbound codepaths remain.
- Approval invalidation logic can over-trigger and hurt UX if field classification is wrong.
- Policy lineage gaps can break auditability claims.
- Async dispatch race conditions can produce duplicate sends without strict idempotency checks.
