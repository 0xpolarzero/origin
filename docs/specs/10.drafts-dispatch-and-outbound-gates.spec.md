# 10 Spec: Draft Dispatch and Outbound Safety Gates

## Goal

Implement a generic Draft and Dispatcher foundation for outbound integrations so agents can author adapter-specific payloads without transport knowledge, while all outbound writes remain explicit, revalidatable, exactly-once, and auditable.

## Why This Phase Exists

The earlier Phase 10 draft assumed a generic outbound system without defining the contract for heterogeneous integrations such as Telegram, GitHub, Slack, and Discord. That gap makes `draft`, `edit`, `approve`, and `send` ambiguous. This phase defines the common envelope that all outbound actions must use, while leaving adapter-specific payload shapes and transport details to backend-owned adapters.

## In Scope

- Generic outbound contract:
  - `Draft` envelope fields:
    - `source_kind`
    - `adapter_id`
    - `integration_id`
    - `action_id`
    - `target`
    - `payload_json`
    - `payload_schema_version`
    - `preview_text`
    - `material_hash`
    - `block_reason_code`
  - adapter-owned action registry
  - per-action payload schema validation
  - adapter-owned action/dispatch registry keyed by `adapter_id` + `action_id` + `payload_schema_version`
  - integration-instance configuration constraining enabled-state, auth, and allowed `target` values within that adapter contract
  - protocol-specific egress matchers hidden behind adapters
- Separate outbound `dispatch_attempt` identity and persistence.
  - This is distinct from the Phase `08` JJ reconciliation `integration_attempt`.
  - required fields:
    - `dispatch_attempt_id`
    - `draft_id`
    - `idempotency_key`
    - `state`
    - `remote_reference`
  - lifecycle:
    - `created -> dispatching -> remote_accepted -> finalized|failed|blocked`
  - restart/recovery semantics must reuse the same `idempotency_key` when retrying/finalizing one logical send.
- Draft lifecycle UI/runtime:
  - `pending`, `blocked`, `approved`, `auto_approved`, `sent`, `rejected`, `failed`
- Pending/Processed tab semantics:
  - Pending: `pending`, `blocked`, `approved`, `auto_approved`
  - Processed: `sent`, `rejected`, `failed`
- Explicit user actions:
  - approve (ready-to-send)
  - send now
  - reject
  - edit
- Material edit semantics:
  - any change to `source_kind`, `adapter_id`, `integration_id`, `action_id`, `target`, or `payload_json` is material
  - material edits invalidate approval and require a fresh decision
  - optional future annotations such as tags/notes are non-material unless explicitly promoted later
- Dispatch-time revalidation:
  - payload schema version
  - action/target constraints
  - integration enabled-state
  - auth health
  - policy
- Strict runtime outbound gate:
  - managed integration endpoints accept writes only from dispatcher with valid `draft_id` + `dispatch_attempt_id`
  - managed endpoint inventory is explicitly declared by adapters
- Non-origin outbound hard rejection (`workspace_policy_blocked`) with user-visible remediation outcome.
- Auto-approve default OFF and policy default-deny behavior.
- Policy lineage fields persisted for allow/deny/dispatch decisions.
- At least one deterministic adapter-owned action path for end-to-end validation in this phase.
  - This may be a test/system adapter.
  - Reserve `adapter_id=test` for the deterministic validation adapter in this phase.
  - Reserve `action_id=message.send` and `action_id=issue.create` as the minimum required incompatible test actions for schema and lifecycle coverage.
  - Real Telegram/GitHub/Slack/Discord production adapters are not required in this phase.

## Out of Scope

- Cron/signal scheduling mechanics (Phase `11`).
- Debug reminder flow (Phase `12`).
- Governance/privacy hardening beyond the baseline dispatcher contract (Phase `13`).
- Shipping production-quality adapters for Telegram, GitHub, Slack, Discord, or similar platforms.
- Rich adapter-specific compose UIs beyond the generic Draft review/edit surface.
- Backward compatibility with the current thin Draft schema or preservation/backfill of pre-release Draft data.

## Baseline Assumptions

- Not all outbound integrations share one payload shape; the common model is an envelope plus adapter-owned action schemas.
- Agents author `adapter_id` + `action_id` + `target` + `payload_json`; they do not own transport/auth details.
- Backend adapters own payload validation, preview generation, managed endpoint inventory, and `dispatch()` behavior; integration instances provide configured auth and target allowlists within that adapter contract.
- One deterministic test/system adapter can prove the contract end-to-end without relying on external SaaS availability.
- The deterministic validation adapter IDs are fixed in this phase as `adapter_id=test`, with required test actions `message.send` and `issue.create`.
- One run may emit zero, one, or many outbound Drafts; Draft lifecycle and dispatch identity are per-Draft, not per-Run.
- Phase `08` `integration_attempt` remains JJ reconciliation/apply state only; outbound dispatch introduces a separate `dispatch_attempt`.
- Pre-release runtime data does not need preservation in this phase; replacing the current thin Draft shape is preferred over compatibility shims or legacy backfills.
- Draft rows, audit lineage fields, and history surfaces exist or can be replaced or extended in this phase.
- API authorization boundaries for `approve` vs `send now` are enforceable at runtime entrypoints.
- Dispatcher can atomically validate draft/workspace/integration/attempt bindings before write.

## Agent Execution Plan

### Step 1: Define the Generic Outbound Contract

- Add the canonical Draft envelope fields.
- Replace the current thin Draft schema directly; do not preserve backward compatibility with pre-redesign Draft rows or APIs unless a later spec explicitly requires it.
- Define adapter-owned action registry, per-action payload schema contract, and adapter-owned action/dispatch registry keyed by `adapter_id` + `action_id` + `payload_schema_version`.
- Introduce `dispatch_attempt` as a distinct outbound identity with lifecycle, idempotency key, and recovery semantics.
- Keep JJ reconciliation `integration_attempt` semantics untouched.

### Step 2: Implement Draft Lifecycle and Read Models

- Add deterministic Draft create/edit/approve/reject/send services.
- Extend history/UI read models for Drafts with Pending/Processed tabs.
- Ensure `approved` and `auto_approved` remain in Pending until dispatch reaches a terminal outcome.

### Step 3: Implement Material Edit Invalidation

- Recompute `material_hash` on every material edit.
- Reset `approved` and `auto_approved` Drafts to `pending` on material changes.
- Preserve auditability of invalidation reasons.

### Step 4: Implement Dispatcher and Revalidation

- Add centralized dispatcher entrypoint for managed outbound writes.
- Revalidate schema version, action/target constraints, integration enabled-state, auth health, and policy at send time.
- Reuse the same `dispatch_attempt` idempotency key when recovering or retrying one logical send.
- Fail closed to `blocked` or `failed` with deterministic reason/failure codes when revalidation fails.

### Step 5: Implement Strict Managed Endpoint Gating

- Require valid `draft_id` + `dispatch_attempt_id` for managed outbound writes.
- Reject direct or forged writes to managed endpoints.
- Enforce Origin-only outbound policy for this phase.
- Make the adapter-owned action/dispatch registry and target inventory fail closed for unregistered actions, schema versions, and targets.

### Step 6: Prove the Contract with a Deterministic Adapter

- Add one deterministic adapter-owned action path in tests and local e2e.
- Implement the reserved validation adapter as `adapter_id=test`.
- Implement the reserved incompatible validation actions as `action_id=message.send` and `action_id=issue.create`.
- Use it to validate the full Draft -> approve -> send -> audit flow.
- Cover both user-authored outbound Drafts and system-authored Drafts such as reporting actions.
- Do not depend on real third-party SaaS services in this phase.

### Step 7: Validation Coverage

- adapter/action schema validation
- lifecycle transitions
- edit invalidation
- revalidation failures
- direct-call and forged-context rejection
- non-origin hard-block
- exactly-once dispatch behavior
- remote-accepted crash/restart recovery

## Acceptance Criteria

- `AC-01` Generic outbound Drafts are represented as adapter-owned action envelopes, not a single shared payload shape.
  - Verification: schema/unit tests create Drafts for at least two actions with incompatible payload schemas and inspect persisted fields.
  - Pass condition: canonical fields (`source_kind`, `adapter_id`, `integration_id`, `action_id`, `target`, `payload_json`, `payload_schema_version`, `preview_text`, `material_hash`, `block_reason_code`) are persisted or deterministically derived; valid action payloads succeed; cross-action payloads fail deterministically.

- `AC-02` Outbound dispatch uses a `dispatch_attempt` identity distinct from the Phase `08` JJ `integration_attempt`.
  - Verification: schema/service tests dispatch a Draft and inspect persisted attempt records.
  - Pass condition: outbound writes validate `dispatch_attempt_id`; persisted attempt state follows the documented lifecycle; one logical retry/recovery reuses the same `idempotency_key`; JJ reconciliation attempt state remains separate and is not reused as the generic outbound contract.

- `AC-03` Approve and send are separate actions.
  - Verification: UI/service tests.
  - Pass condition: `approve` does not dispatch; `send now` dispatches only from `approved` and `auto_approved`.

- `AC-04` Material edits invalidate approval.
  - Verification: approve a Draft, edit `action_id`, `target`, or `payload_json`, then inspect state.
  - Pass condition: state resets to `pending` with deterministic invalidation reason and requires re-approval.

- `AC-05` Dispatch revalidation fail-closes.
  - Verification: change payload schema version, revoke auth, disable integration, or change policy after approval.
  - Pass condition: dispatch becomes `blocked` or `failed` with explicit reason/failure code and no external write.

- `AC-06` Direct managed outbound calls without Draft context are rejected.
  - Verification: adapter endpoint test without valid `draft_id` + `dispatch_attempt_id`.
  - Pass condition: call is rejected and audit evidence is emitted.

- `AC-07` Forged context mismatches are rejected.
  - Verification: dispatch attempts with valid-looking but mismatched `draft_id`, `dispatch_attempt_id`, workspace, or integration bindings.
  - Pass condition: request is rejected with deterministic policy error and no outbound write.

- `AC-08` Non-origin outbound attempts are rejected and visible.
  - Verification: attempt outbound action in a standard workspace.
  - Pass condition: deterministic `workspace_policy_blocked` outcome is recorded and surfaced with remediation hint.

- `AC-09` Policy lineage fields are persisted for allow/deny/dispatch decisions.
  - Verification: audit/event query tests.
  - Pass condition: `policy_id`, `policy_version`, `decision_id`, and `decision_reason_code` are present and immutable for those events.

- `AC-10` Revalidation failures produce zero side effects.
  - Verification: induce revalidation failure after approval and before adapter dispatch.
  - Pass condition: no external write occurs and deterministic blocked/failure code is recorded.

- `AC-11` Concurrent `send now` requests are exactly-once.
  - Verification: concurrent `send now` requests and restart/retry race tests.
  - Pass condition: at most one outbound side effect occurs for one logical Draft send.

- `AC-12` Adapter/action registry and target inventory are fail-closed.
  - Verification: attempt outbound dispatch with unregistered adapter/action/schema-version combination or `target` outside the integration-instance allowlist and adapter-managed inventory.
  - Pass condition: request is rejected with deterministic policy error and no outbound write occurs.

- `AC-13` Policy-evaluation failure at decision time is default-deny.
  - Verification: force policy evaluation failure before approval/auto-approval decision.
  - Pass condition: Draft transitions to `blocked` and no dispatch is attempted.

- `AC-14` Dispatch recovery is deterministic across `remote_accepted -> finalize` crash boundaries.
  - Verification: crash-injection tests after remote acceptance and before local finalize, followed by restart/recovery.
  - Pass condition: the same `dispatch_attempt`/`idempotency_key` is finalized without duplicate outbound side effects.

- `AC-15` A deterministic adapter-owned action path proves the full contract without external SaaS dependency.
  - Verification: local integration/e2e tests exercise Draft create/edit/approve/send against the reserved `adapter_id=test` adapter and its `message.send` action; schema validation tests also cover the incompatible `issue.create` action.
  - Pass condition: the end-to-end flow passes using backend-owned adapter dispatch with no real third-party dependency, and the reserved action IDs are used consistently.

- `AC-16` Validation commands pass.

## Risk Register

- The generic envelope can become too weak if action schemas are underspecified.
- Overloading JJ reconciliation concepts into outbound dispatch will create long-term confusion if naming is not kept separate.
- Adapter schema/version drift can break unfinished implementation work if action ownership is unclear.
- Strict dispatcher gating can expose stale pre-release codepaths that should be deleted instead of preserved.
- A deterministic test adapter can prove the contract shape but may miss SaaS-specific quirks that later adapter phases must cover.
