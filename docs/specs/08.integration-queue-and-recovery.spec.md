# 08 Spec: JJ Integration Queue and Crash-Safe Recovery

## Goal

Implement serialized per-workspace JJ reconciliation/apply integration with immutable queue ordering, idempotent integration attempts, stale-base replay, and deterministic cancel/restart outcomes.

## Why This Phase Exists

Manual execution exists after Phase 07, but checkpoint JJ reconciliation/apply safety is the highest-risk part of correctness. This phase isolates JJ queue/recovery semantics before the later generic Draft/Dispatcher outbound architecture and automated triggers are added.

## In Scope

- Per-workspace JJ integration serialization lock.
- FIFO queue by immutable `ready_for_integration_at` + `run_id` tiebreaker.
- Integration attempt lifecycle with `(run_id, integration_attempt_id)` uniqueness for JJ apply/reconciliation.
- Atomic integration boundary enforcement:
  - JJ mutation + DB linkage commit
- Crash recovery behavior:
  - recover in-flight attempt
  - finalize post-mutation pre-DB crash path
- Stale-base detection and one replay attempt (`replay_index`).
- Deterministic cancellation semantics during integrating/reconciling.
- Failure-code emission for replay exhaustion and cleanup failures.

## Out of Scope

- Generic Draft/Dispatcher outbound send flow.
- Cron/signal trigger creation.
- Long-running debug reminder UX.

## Baseline Assumptions

- Phase 04 state models and Phase 05 JJ adapter are available.
- Phase 07 creates integration-ready run outputs.
- Run queue persistence is DB-backed and recoverable on startup.
- Locking model is valid under multi-process contenders on the same workspace storage.
- Crash-injection hooks are available for post-apply/pre-finalize recovery tests.

## Agent Execution Plan

### Step 1: Implement Queue Persistence and Ordering

- Persist immutable `ready_for_integration_at` once on enqueue.
- Enforce deterministic dequeue ordering.
- Preserve ordering across retries and restarts.

### Step 2: Implement Workspace Integration Locking

- Add per-workspace lock acquisition/release path.
- Ensure only one integrating run per workspace at a time.

### Step 3: Implement Attempt Identity and Atomic Boundary

- Create `integration_attempt_id` before apply.
- Persist attempt state transitions:
  - `attempt_created`
  - `jj_applied`
  - `db_linked`
  - `finalized`
- Enforce atomic commit boundary:
  - apply JJ mutation
  - commit DB operation linkage

### Step 4: Implement Recovery and Replay

- On startup:
  - detect unfinished attempts
  - recover/finalize without duplicate mutation
- Detect stale-base via `integration_head_change_id_before_apply` mismatch.
- Replay sequence is deterministic: detect mismatch -> `workspace update-stale` -> reapply -> increment `replay_index`.
- If replay fails/exhausted, set `stale_base_replay_exhausted`.

### Step 5: Implement Cancel Edge Semantics

- Pre-integration cancel => `canceled` and queue removal.
- In integration/reconcile:
  - mark `cancel_requested`
  - finish current atomic step
  - resolve to `completed` if committed, else `canceled`.
- Timeout/cancel race: first terminal write wins.

### Step 6: Test Matrix

- concurrency ordering
- crash injection points
- replay path
- cancel race path
- cross-workspace parallelism
- stale lock recovery on startup

## Acceptance Criteria

- `AC-01` Queue ordering is immutable under retries/restarts.
  - Verification: integration queue tests with forced retries/restarts.
  - Pass condition: dequeue order remains based on initial key.

- `AC-02` Only one integrating run per workspace is active at any time.
  - Verification: concurrent run test in same workspace.
  - Pass condition: lock prevents parallel integration.

- `AC-03` Crash after JJ mutation before DB finalize does not double-apply.
  - Verification: crash-injection test at post-mutation checkpoint.
  - Pass condition: exactly one effective mutation and one finalized operation linkage row for the changed run.

- `AC-04` Stale-base replay occurs at most once.
  - Verification: force head drift, then drift again.
  - Pass condition: one replay attempt then terminal `stale_base_replay_exhausted`.

- `AC-05` Cancel during integration follows deterministic outcome contract.
  - Verification: cancel timing matrix tests.
  - Pass condition: committed integration resolves `completed` + marker; uncommitted resolves `canceled`.

- `AC-06` Cross-workspace integrations can progress in parallel while same-workspace integrations stay serialized.
  - Verification: multi-workspace concurrency tests.
  - Pass condition: no parallel integrating runs in one workspace, but parallel progress across distinct workspaces.

- `AC-07` Startup recovers stale/incomplete lock state deterministically.
  - Verification: forced stale lock scenario on startup.
  - Pass condition: stale lock is recovered and queue processing resumes deterministically.

- `AC-08` Cancel-timeout race resolves to exactly one terminal state.
  - Verification: race-condition test at timeout boundary with concurrent cancel request.
  - Pass condition: one terminal state wins and losing event is recorded as no-op.

- `AC-09` Validation commands pass.

## Risk Register

- Locking bugs can deadlock integration processing.
- Recovery path mistakes can silently duplicate mutations.
- Replay heuristics may misclassify drift cause.
- Cancel race handling can regress if terminal precedence is not centralized.
