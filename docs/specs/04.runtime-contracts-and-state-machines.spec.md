# 04 Spec: Runtime Contracts and State Machines

## Goal

Establish deterministic runtime contracts for `Run`, `Operation`, `Draft`, and audit/event records so all later phases build on a single, testable persistence model.

## Why This Phase Exists

The PRD now defines strict lifecycle and policy behavior. Implementation should not start with ad hoc state updates. We need a contract-first baseline that guarantees:

- canonical statuses and failure codes
- legal transition edges
- idempotent attempt identities
- auditable event lineage

Without this phase, later queue/integration/draft features will diverge and become hard to validate.

## In Scope

- Add/extend DB schema for:
  - `run` with canonical status enum and reason/failure fields
  - `operation` with linkage fields and status
  - `draft` lifecycle fields and provenance
  - `integration_attempt` identity and replay fields
  - audit/event table with required baseline fields
- Add enum and transition-guard layer in runtime services.
- Add canonical failure/reason code registry from PRD plus phase-scoped extensions. Later phases may not introduce new code literals without adding them to this registry first.
- Enforce illegal-transition rejection at service layer.
- Add deterministic indexes/constraints:
  - uniqueness for `(run_id, integration_attempt_id)`
  - immutable queue ordering key behavior for `ready_for_integration_at`
  - dedupe-oriented lookup indexes needed by later phases
  - policy/dispatch lineage lookup indexes on audit/event records
- Add migration(s) and startup schema compatibility checks.
- Add SDK/runtime types if server contracts are changed.

Canonical schema contract in this phase must explicitly list:

- columns per entity (`run`, `operation`, `draft`, `integration_attempt`, `audit_event`)
- constraints (unique, nullable, default, foreign-key expectations)
- indexes and query intent

Baseline column/constraint floor for this phase:

- `run`:
  - `id` (uuidv7, pk)
  - `status` (enum, indexed)
  - `trigger_type`
  - `workspace_id`
  - `session_id`
  - `ready_for_integration_at` (immutable after first set)
  - `failure_code` / `reason_code`
  - timestamps (`created_at`, `updated_at`, `started_at`, `finished_at`)
- `operation`:
  - `id` (uuidv7, pk)
  - `run_id` (fk)
  - `workspace_id`
  - `status`
  - `source_operation_id` (nullable fk for revert linkage)
  - JJ linkage fields from PRD
- `draft`:
  - `id` (uuidv7, pk)
  - `run_id` (nullable fk)
  - `workspace_id`
  - `status`
  - `integration_id` (baseline outbound binding identifier; later phases may widen Draft envelope with adapter/action/target/payload fields without changing this lifecycle contract)
  - policy lineage fields
  - timestamps
- `integration_attempt`:
  - `id` (`integration_attempt_id`, pk)
  - `run_id`
  - `workspace_id`
  - `state` (`attempt_created`, `jj_applied`, `db_linked`, `finalized`)
  - `replay_index`
  - unique `(run_id, id)`
- `audit_event`:
  - `id` (uuidv7, pk)
  - `event_type`, `actor_type`, `occurred_at`
  - linkage IDs (`workspace_id`, `run_id`, `operation_id`, `draft_id`, `integration_attempt_id`; later outbound phases may add `dispatch_attempt_id` for outbound dispatch provenance)
  - policy lineage fields (required for policy/dispatch events)
  - typed `event_payload`

Clarification for later phases:

- Phase `04` defines the baseline Draft lifecycle and JJ reconciliation attempt contract.
- Later outbound phases may add a wider Draft envelope and a distinct outbound `dispatch_attempt` identity, including audit-event linkage for outbound dispatch provenance, without changing the state-machine rules established here.

- Drizzle/schema system remains the source of truth for runtime tables.
- Existing runtime can accept additive migrations without destructive reset.
- Runtime IDs in this phase use UUIDv7 only.
- Transition write + event write for one state change happen in one transaction boundary.
- Direct table writes outside transition services are removed or blocked in touched paths.

## Agent Execution Plan

### Step 1: Add Canonical Runtime Enums and Failure Codes

- Introduce shared enums for:
  - `run.status`
  - `operation.status`
  - `draft.status`
  - standardized failure/reason codes
- Ensure enums are reused by DB schema + service validation + API serialization.

### Step 2: Extend DB Schema and Constraints

- Add required columns and indexes for PRD contracts:
  - immutable queue ordering key
  - integration attempt identity and replay fields
  - policy lineage fields on relevant events
- Enforce uniqueness and nullability constraints.
- Provide deterministic default values where required.

### Step 3: Add Transition Guard Service

- Add centralized transition function per entity.
- Reject illegal transitions with machine-readable errors.
- Record mandatory terminal metadata (`reason`, `failure_code`, timestamps).
- Check legal edges against an explicit in-spec transition matrix and keep matrix synced with tests.

Canonical legal transition matrix in this phase:

- `run.status` creation semantics:
  - `create -> queued` for executable triggers.
  - `create -> skipped` allowed only for scheduler-determined non-executable slots (for example missed offline windows or DST spring-forward gaps) and requires non-null `reason_code`.
  - `create -> skipped` rows are terminal on insert; no session execution is started.
- `run.status`:
  - `queued -> skipped` is forbidden (skipped outcomes are direct-create only in this phase contract)
  - `queued -> running|canceled`
  - `running -> validating|failed|canceled`
  - `validating -> ready_for_integration|completed_no_change|failed|canceled`
  - `ready_for_integration -> integrating|canceled`
  - `integrating -> reconciling|completed|failed|cancel_requested`
  - `reconciling -> integrating|failed|cancel_requested`
  - `cancel_requested -> completed|canceled|failed`
  - `completed|completed_no_change|failed|canceled|skipped` are terminal
- `operation.status`:
  - `completed -> reverted`
  - `reverted` terminal
- `draft.status`:
  - `pending -> approved|auto_approved|blocked|rejected`
  - `approved|auto_approved -> sent|failed|blocked|pending` (pending on material edit invalidation)
  - `blocked -> pending|rejected|failed`
  - `sent|rejected|failed` terminal

### Step 4: Add Audit/Event Baseline Writer

- Add typed event creation utility that validates required fields by event category.
- Enforce schema-level redaction rules for payload shape (no secret-like free-form fields in baseline typed payload).

### Step 5: Validation and Regression Coverage

- Add unit tests for state transitions and forbidden edges.
- Add migration tests for schema evolution.
- Add persistence tests for uniqueness/constraint enforcement.
- Add migration/backfill tests for legacy rows missing newly required fields.

## Acceptance Criteria

- `AC-01` Canonical enums exist and are reused across schema and runtime state transitions.
  - Verification: unit tests import the same enum source in DB/runtime modules.
  - Pass condition: no duplicated ad hoc enum strings in transition logic.

- `AC-02` Illegal transitions are rejected deterministically.
  - Verification: unit tests attempt invalid edges for `run`, `operation`, `draft`.
  - Pass condition: each invalid transition returns machine-readable error and no DB mutation.

- `AC-03` Full legal-edge coverage exists for canonical transition matrix.
  - Verification: transition tests enumerate all legal edges and creation semantics, including `create -> skipped` constraints.
  - Pass condition: no legal edge is untested, `queued -> skipped` is rejected, and no undocumented edge/path is allowed.

- `AC-04` Required uniqueness and immutability constraints are enforced.
  - Verification: persistence test inserts duplicate `(run_id, integration_attempt_id)`.
  - Pass condition: duplicate insert fails and attempts to mutate immutable queue key fail.

- `AC-05` Required audit baseline fields and payload schema/redaction constraints are enforced for applicable event types.
  - Verification: tests attempt writes missing required policy lineage fields for policy/dispatch events.
  - Pass condition: invalid writes are rejected with explicit validation error and secret-like payload fields are blocked.

- `AC-06` Migration/backfill is forward-applicable on existing local DB state.
  - Verification command:
```bash
bun run --cwd packages/opencode test -- storage
```
  - Pass condition: migration/backfill tests pass and no destructive reset is required.

- `AC-07` Typecheck/test baseline for touched packages is green.
  - Verification commands:
```bash
bun run --cwd packages/opencode typecheck
bun run --cwd packages/opencode test
bun run --cwd packages/app typecheck
```
  - Pass condition: all commands pass.

## Risk Register

- Schema churn may require follow-up migrations if later phases discover missing fields.
- Overly strict constraints can block transitional writes during rollout.
- Poor event payload typing can create frequent migration noise.
- Transition guards can be bypassed if direct table writes remain in legacy paths.
