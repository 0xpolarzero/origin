# 15 Spec: Graph-First Workflow Orchestration

## Goal

Replace Origin's session-first workflow authoring and execution assumptions with a graph-first workflow model that preserves the current desktop/runtime foundations while adding workflow revisions, immutable run snapshots, graph-native run/read models, workflow edit history, and dedicated workflow/run/library surfaces.

## Why This Phase Exists

The current workflow direction across earlier phases is still anchored to a session-first mental model:

- Phase `06` defines YAML workflows and library resources, but still frames workflows mainly as trigger + steps + references.
- Phase `07` defines manual execution as one new session per top-level run.
- Phase `09` exposes runs and operations primarily through history and session navigation rather than through a workflow graph and run page.

That model is no longer sufficient for the clarified product direction. Workflows now need to behave like real orchestration graphs:

- AI-first creation and refinement
- direct graph editing
- workflow-local and shared reusable resources
- multiple agent requests per workflow
- script and agent nodes rendered side by side in one graph
- immutable run snapshots
- deterministic partial reruns
- linked sessions as detail artifacts rather than as the workflow container

This phase exists to make that model real before later trigger, debug, and release-gate work continue on top of outdated workflow assumptions.

## In Scope

- Graph-native runtime objects and read models:

  - `builder`

## Out of Scope

- Interactive approval or input-request nodes inside workflows
- Subworkflows
- Runtime-generated graph topology
- Visual rule-builder authoring for conditions
- Cloud execution, multi-user collaboration, or non-desktop scope
- Full sandboxing and permission-profile enforcement
- Release-gate hardening work already described in later phases, except where this phase changes their workflow assumptions

## Baseline Assumptions

- The desktop-first, local-first, JJ-backed, DB-backed direction from Phases `01` through `10` remains in force.
- `save == live` for workflow definitions in v1. There is no separate draft/publish workflow-definition mode.
- One workflow edit session may publish multiple live revisions over time.
- Runs start from live workflow revisions only in v1. Checkpoints are nested edit-session artifacts, not alternate runnable drafts.
- Every run snapshot freezes exact shared-library content alongside the workflow revision and workflow-local resources.
- The app remains session-first overall; only dedicated workflow and run surfaces become graph-first.
- Full run-frame persistence on every scheduler tick is not required in v1. Persist run, node, attempt, and event records; derive most frames on read unless milestone frames are required.
- Workflow edit sessions are intent-scoped to exactly one workflow but are not sandboxed to workflow files only. They may touch arbitrary workspace files, and workflow edit history must surface that fact explicitly.
- Workflow runs do not rewrite workflow definitions or library items by default.
- Deleting a workflow with historical runs should behave like archive/hide rather than erasing historical run surfaces.
- Shared library items that are still referenced by workflows should be blocked from deletion by default.
- Non-terminal workflow runs auto-resume after restart from persisted run state.

Specific earlier assumptions that this phase explicitly supersedes:

- Phase `06` no longer treats workflows merely as trigger + steps + library references; workflows are graph-first orchestration definitions with local and shared resources, revisions, and snapshots.
- Phase `07` no longer requires one top-level chat session per run; the run is the execution container and sessions are linked artifacts by role.
- Phase `07` no longer defines repair behavior as "same run session thread"; retries, validation, and reconcile are represented in graph-native runtime state and linked sessions where applicable.
- Phase `09` no longer treats session deep links as the primary execution navigation target; workflow and run pages become the primary surfaces.

Future planned phases `11` through `14` must inherit this graph-first workflow model where they touch triggers, debug flows, history, or release validation.

## Required Keep Set

- Keep the canonical workflow and library definitions on disk as YAML-backed user-visible files.
- Keep the current top-level runtime entities:
  - `Run`
  - `Operation`
  - `Draft`
  - `IntegrationAttempt`
  - `DispatchAttempt`
  - `AuditEvent`
- Keep the Draft/Dispatcher outbound safety model.
- Keep the app-wide session surface for normal chat usage.
- Keep existing session/chat, diff, terminal, and history components where they can be reused inside workflow and run detail views.

## Deletion Candidates

Not applicable. This phase replaces workflow assumptions and surfaces but does not require repo pruning or destructive product-scope removal.

## Agent Execution Plan

### Step 1: Establish Workflow Revision and Snapshot Contracts

- Add workflow-native runtime contracts for:
  - workflow revisions
  - workflow edit sessions
  - checkpoints
  - immutable run snapshots
- Ensure run snapshots freeze:
  - workflow revision
  - workflow-local resources
  - exact shared-library content
  - run input and trigger metadata
- Keep existing top-level runtime entities intact and extend them rather than replacing them.

### Step 2: Introduce Graph-Native Runtime Objects

- Add runtime objects and read models for:
  - run node
  - run attempt
  - run event
  - milestone run frame if needed
- Define deterministic node and attempt status transitions compatible with the existing runtime transition philosophy.
- Treat branch, parallel, loop, validation, and reconcile as first-class graph-visible execution items.

### Step 3: Replace Session-First Workflow Execution Assumptions

- Remove the requirement that one top-level run equals one main session thread.
- Link sessions to workflows, runs, and nodes by explicit role instead of relying on `parent_id`.
- Ensure execution-node transcripts remain available as detailed artifacts without polluting the global sidebar.
- Make execution-node transcripts read-only by default and expose explicit `Continue from here`.
- Keep run-followup analysis/debug-first and fork a dedicated workflow edit session when the user chooses to apply changes back to the workflow.

### Step 4: Build Graph-First Workflow and Run Surfaces

- Add:
  - workflow detail route
  - run detail route
  - library item detail route
- Make the workflow page graph-primary with chat docked into it.
- Make the run page graph + summary first, with deep-linkable node detail panels.
- Render the full workflow snapshot graph in run view and keep unreached/skipped paths visible but dimmed with explicit reasons.
- Reuse existing chat, diff, terminal, and detail components where appropriate.

### Step 5: Implement AI-First Authoring and Direct Graph Editing

- Add the workflow builder flow that starts from an AI-guided session.
- Ensure AI edits and direct graph edits both modify the same canonical workflow files.
- Support basic direct topology edits in the graph UI:
  - add node
  - delete node
  - reconnect edges
  - edit node settings
- Support workflow-local resource creation and shared-library promotion.
- Support typed manual-run inputs using the v1 input set declared in scope.

### Step 6: Implement History, Visibility, and Deep-Link Changes

- Add `Workflow Edits` to global history.
- Group top-level workflow edit history by workflow edit session, not by save.
- Nest checkpoints inside the workflow edit session detail.
- Route notifications and deep links to workflow and run surfaces rather than hidden execution sessions.
- Attribute operations and drafts to specific run nodes where possible.
- Surface non-workflow workspace-file changes explicitly when they occur inside workflow edit sessions.

### Step 7: Implement Deterministic Rerun and Parallel-Reconcile Semantics

- Add `Rerun workflow` and `Rerun from here`.
- Ensure `Rerun from here` creates a new run, preserves valid upstream results, invalidates downstream dependents, and reuses the same snapshot and input.
- Execute parallel branches in isolated branch workspaces from the same base snapshot.
- Add implicit join behavior with one reconcile pass and deterministic failure when reconcile cannot resolve conflicts.

### Step 8: Align Future-Phase Assumptions and Add Validation Coverage

- Update planning docs and acceptance references where needed so future trigger/debug/governance/release-gate implementation work consumes the graph-first model instead of older session-first assumptions.
- Add or update unit, integration, and local e2e coverage for:
  - workflow revision and snapshot behavior
  - graph execution and rerun semantics
  - session role visibility
  - history grouping and deep linking
  - parallel reconcile behavior

## Acceptance Criteria

- `AC-01` Workflow source-of-truth behavior is graph-first and canonical.
  - Verification: create, edit, and reload workflows through AI flow, direct graph edits, and manual file edits.
  - Pass condition: all surfaces converge on the same workflow YAML and sibling resource files, and file edits remain the canonical source of truth.

- `AC-02` Workflow revisions, checkpoints, and run snapshots are distinct and immutable in the documented way.
  - Verification: create multiple workflow saves inside one edit session, inspect nested checkpoints, then run and rerun the workflow after later edits.
  - Pass condition: workflow edit history groups by session, checkpoints remain nested, each run references an immutable snapshot, and older runs still render the graph they actually executed.

- `AC-03` Runs no longer require one top-level run session and instead use explicit session-link roles.
  - Verification: execute workflows with agent nodes, scoped node edits, and run follow-up chats.
  - Pass condition: builder, node-edit, execution-node, and run-followup sessions behave according to role visibility rules, do not rely on `parent_id` to drive workflow semantics, execution-node transcripts are read-only by default with explicit `Continue from here`, and applying workflow changes from run-followup forks a dedicated workflow edit session.

- `AC-04` Workflow and run pages are the primary graph-first surfaces.
  - Verification: local e2e coverage for workflow detail, run detail, library detail, node deep links, and side-panel behavior.
  - Pass condition: workflow pages default to `Design`, run pages default to graph + summary, run pages render the full workflow snapshot with dimmed unreached or skipped paths, node panels are URL-addressable, and library items expose `Used By` and history.

- `AC-05` Graph execution status and rerun semantics are deterministic.
  - Verification: unit and integration tests for run-node status transitions, run-attempt transitions, skipped-path reasons, `Rerun workflow`, and `Rerun from here`.
  - Pass condition: node and attempt states follow the approved vocabulary and legal transitions, and `Rerun from here` preserves valid upstream results while invalidating downstream dependents.

- `AC-06` Manual-run inputs and restart behavior match the workflow contract.
  - Verification: local e2e and integration coverage for typed manual-run inputs plus app restart during a non-terminal workflow run.
  - Pass condition: manual runs support the declared typed input set, in-progress runs resume from persisted state after restart, and resumed runs do not adopt newer workflow revisions or library content.

- `AC-07` Parallel mutation behavior is isolated and reconcilable.
  - Verification: integration tests where parallel branches mutate overlapping files and require reconcile.
  - Pass condition: each branch runs in isolation from the same base snapshot, reconcile appears as a visible system node, one reconcile pass runs on conflict, and unresolved reconcile failure is deterministic.

- `AC-08` History and notification behavior are workflow-native without losing existing runtime visibility.
  - Verification: local e2e coverage for `Workflow Edits`, `Runs`, `Operations`, `Drafts`, workflow-scoped history, and run notifications.
  - Pass condition: workflow edit history is one row per edit session with nested checkpoints, run notifications deep-link to run pages, workflow edit sessions surface broader workspace-file changes when they occur, and runs/operations/drafts remain visible with node attribution where available.

- `AC-09` Shared-library and workflow-local resource behavior matches the product contract.
  - Verification: tests for reuse, promotion, `used by`, user copy-on-write dialog behavior, and future-run dependence on shared-library edits.
  - Pass condition: local resources stay workflow-owned, shared items expose impact and usage, future runs use updated shared items, past runs remain frozen through snapshots, and shared items still in use are blocked from deletion by default.

- `AC-10` Existing reusable UI surfaces are preserved where intended.
  - Verification: component and e2e coverage for agent-node transcript rendering, script-node diff/log/output rendering, and follow-up chat entrypoints.
  - Pass condition: the implementation reuses current session/detail surfaces for node drill-downs rather than introducing a second unrelated transcript or artifact system.

- `AC-11` Workflow lifecycle retention rules are preserved.
  - Verification: tests or fixtures for deleting a workflow with historical runs and for duplicating a workflow with local plus shared resources.
  - Pass condition: deleting a workflow with historical runs archives or hides it instead of erasing historical run surfaces, and workflow duplication copies workflow-local resources while preserving shared-library references by default.

- `AC-12` Future trigger/debug/governance implementation is not blocked by session-first assumptions.
  - Verification: planning-doc review plus targeted tests or fixtures showing manual, cron, and signal runs bind to live revisions and historical snapshots consistently.
  - Pass condition: no accepted workflow/run/history surfaces require a main run chat session, and future phases can attach their behavior to workflow/run/node surfaces without contradicting this model.

## Risk Register

- Session-rooted assumptions in the current sidebar, notifications, and permission behavior can leak back into the new workflow model if `parent_id` is reused instead of explicit session links.
- A partial implementation can leave Phase `06`, `07`, and `09` assumptions in conflict, producing mismatched product behavior and documentation.
- Parallel branch isolation and reconcile logic can become nondeterministic if merge behavior is left to prompt guidance instead of runtime control.
- Workflow edit history can become noisy or misleading if checkpoints escape into top-level history or if one session is allowed to span multiple workflows.
- Route and read-model expansion can overcomplicate the UI if graph-first workflow surfaces are not clearly separated from the app's normal session-first shell.
