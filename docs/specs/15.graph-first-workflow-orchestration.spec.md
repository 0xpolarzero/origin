# 15 Spec: Graph-First Workflow Foundations

## Goal

Deliver the first fully implementable graph-first workflow slice on top of the current desktop/runtime foundation.

This phase replaces the Phase `06` flat workflow file contract and the Phase `07` run-session container assumption for workflow execution by adding:

- canonical graph YAML `schema_version: 2`
- workflow-local sibling resources
- immutable workflow revisions
- immutable run snapshots
- graph-native run-node persistence for a limited executable node set
- explicit runtime `session_link` records instead of workflow semantics on `parent_id`
- dedicated workflow and run detail routes as the primary execution surfaces

This phase is intentionally narrower than the earlier draft of Phase `15`. It establishes the storage, runtime, and routing foundation that later phases can extend without requiring another workflow-model rewrite.

## Why This Phase Exists

The repo currently carries three workflow assumptions that are no longer acceptable:

- Phase `06` treats workflows as flat `instructions + resources` YAML.
- Phase `07` treats one run as one top-level mutable chat session.
- Phase `09` treats history rows and session deep links as the primary execution navigation target.

The approved product direction is still graph-first, but the original Phase `15` draft mixed too many concerns into one implementation phase:

- graph file-contract replacement
- workflow authoring/editor sessions
- workflow edit history/checkpoints
- graph runtime persistence
- full advanced node vocabulary
- partial reruns and reconcile
- new workflow/run/library surfaces
- deep-link and notification rewrites

That bundle is too large and too coupled to be implemented safely from the current codebase.

This rewrite narrows Phase `15` to the smallest coherent slice that still changes the product model in the right order:

1. Replace the workflow file contract with a graph-backed canonical model.
2. Persist immutable workflow and run provenance.
3. Move workflow execution semantics off the main run chat session.
4. Add minimal workflow/run detail routes so future phases have stable targets.

AI-first authoring, workflow edit history, library-item detail management, advanced node types, and partial reruns remain approved direction, but they move to later phases that build on this foundation instead of competing with it.

## In Scope

- Replace the runnable workflow file contract with canonical graph YAML `schema_version: 2`.
- Support one initial executable node set:
  - `agent_request`
  - `script`
  - `condition`
  - `end`
- Support one initial runnable trigger type:
  - `manual`
- Support workflow-local sibling resources under `.origin/workflows/<workflow_id>/`.
- Support shared-library references to existing `.origin/library/` items for the resource kinds used in this phase.
- Add immutable runtime provenance objects:
  - `workflow_revision`
  - `run_snapshot`
  - `run_node`
  - `run_attempt`
  - `run_event`
  - `session_link`
- Keep the existing top-level runtime entities and queue/integration model, but place the graph execution layer underneath them.
- Replace run-as-session semantics for graph workflows with explicit runtime-linked sessions:
  - `execution_node`
  - `run_followup`
- Add minimal graph-first user-facing routes:
  - `/:dir/workflows/:workflowId`
  - `/:dir/runs/:runId`
- Retarget history and notifications so run execution opens workflow/run detail surfaces instead of hidden execution sessions.
- Preserve existing reusable transcript, diff, log, and terminal surfaces by embedding them inside the new run-node drill-downs.
- Define explicit deferral boundaries for later phases so Phases `11` through `14` can extend the same workflow model without a second contract rewrite.

### Canonical Workflow YAML Contract

The canonical workflow file remains YAML on disk, but it becomes graph-first and block-structured.

- Canonical workflow file path:
  - `.origin/workflows/<workflow_id>.yaml`
- Canonical workflow-local resource root:
  - `.origin/workflows/<workflow_id>/`
- Shared library root remains:
  - `.origin/library/`
- Manual file edits remain canonical in this phase. There is no alternate authoring store.
- Workflow identity is the YAML `id` field and is independent from the filename.
- Renaming the file must not change workflow identity.
- Raw edge lists are not supported. The canonical graph format is a block tree rooted at `steps`.
- Node IDs are user-visible stable identifiers.
- Node IDs must be unique across the full workflow tree, including nested condition branches.

Required top-level fields:

- `schema_version: 2`
- `id`
- `name`
- `description` optional
- `trigger`
- `inputs`
- `resources`
- `steps`

Example:

```yaml
schema_version: 2
id: review_release
name: Review Release
trigger:
  type: manual
inputs:
  - key: release_tag
    type: text
    label: Release tag
    required: true
resources:
  - id: review_prompt
    source: library
    kind: prompt_template
    item_id: release_review_prompt
steps:
  - id: inspect
    kind: agent_request
    title: Inspect release
    prompt:
      source: resource
      resource_id: review_prompt
    output:
      type: object
      required:
        - requires_fix
      properties:
        requires_fix:
          type: boolean
  - id: gate
    kind: condition
    title: Requires fix?
    when:
      ref: steps.inspect.output.requires_fix
      op: equals
      value: true
    then:
      - id: stop
        kind: end
        title: Stop
        result: failure
    else:
      - id: done
        kind: end
        title: Done
        result: success
```

Required validation outcomes in this phase:

- `schema_version: 1` workflow files are non-runnable with deterministic `schema_version_unsupported`.
- Unsupported node kinds are non-runnable with deterministic `node_kind_unsupported`.
- Duplicate node IDs are non-runnable with deterministic `node_id_duplicate`.
- Invalid typed inputs are non-runnable with deterministic `input_shape_invalid`.
- Duplicate input keys are non-runnable with deterministic `input_key_duplicate`.
- Invalid manual-input references inside `agent_request` prompt interpolation are non-runnable with deterministic `input_ref_invalid`.
- Broken workflow-local resource references are non-runnable with deterministic `local_resource_missing`.
- Workflow-local resource paths outside `.origin/workflows/<workflow_id>/` are non-runnable with deterministic `local_resource_outside_workflow`.
- Invalid condition references or unsupported condition operands are non-runnable with deterministic `condition_ref_invalid`.

### Supported Trigger Contract

Only manual workflows are runnable in this phase.

Required trigger shape:

- `trigger.type: manual`

Cron and signal triggers are intentionally deferred to Phase `11`. They may extend `schema_version: 2` later; this phase does not pre-commit to a second workflow-file version for that extension.

### Supported Node Contracts

Every authored node must include:

- `id`
- `kind`
- `title`

Supported authored node kinds in this phase:

- `agent_request`
- `script`
- `condition`
- `end`

Deferred node kinds in this phase:

- `parallel`
- `loop`
- `validation`
- `draft_action`

Deferred node kinds must fail validation with `node_kind_unsupported`. They are reserved for additive follow-on phases and must not silently fall back to flat-session behavior.

#### `agent_request`

- Represents exactly one agent request.
- Must specify exactly one prompt source:
  - `prompt.source: inline` with inline prompt text, or
  - `prompt.source: resource` with `resource_id` referencing a `kind: prompt_template` resource.
- After resolving the prompt source, runtime must perform exactly one manual-input interpolation pass.
- Supported placeholder syntax in this phase is `{{inputs.<key>}}`.
- `<key>` must reference a declared manual workflow input.
- Interpolation is literal and non-recursive. Inserted values are not re-parsed for placeholders.
- Runtime must not inject undeclared inputs or any other implicit prompt context.
- Interpolation value rules:
  - `text`, `long_text`, `select`: exact captured value
  - `number`: canonical decimal string
  - `boolean`: `true` or `false`
  - `path`: absolute path to the frozen snapshot copy
- If a referenced input has no captured value in the immutable `run_snapshot`, run start must be rejected before the first node attempt begins. No empty-string fallback is allowed.
- May declare one structured output contract under `output`.
- When `output` is declared, runtime must validate the final structured result and persist it as node output.
- `condition.when.ref` may target only declared scalar fields inside a successful `agent_request` structured output.

#### `script`

- Executes one script or shell command inside the run workspace.
- Must specify exactly one command source:
  - `script.source: inline` with inline command text, or
  - `script.source: resource` with `resource_id` referencing a `kind: script` resource.
- Runtime must not interpolate manual inputs into inline command text, script resources, or `cwd`.
- Each captured manual input is exported to the child process environment as `ORIGIN_INPUT_<KEY>`, where `<KEY>` is the declared input key uppercased.
- Export value rules:
  - `text`, `long_text`, `select`: exact captured value
  - `number`: canonical decimal string
  - `boolean`: `true` or `false`
  - `path`: absolute path to the frozen snapshot copy
- Optional inputs with no captured value are not exported.
- For `path` inputs, the original selected path is audit metadata only and must not be exported to script execution.
- May specify `cwd` relative to the run workspace root.
- The persisted script output envelope must include:
  - `exit_code`
  - `stdout`
  - `stderr`
  - `changed_paths`
- `condition.when.ref` may target `script` output scalar fields.

#### `condition`

- Encodes one deterministic branch decision.
- Must contain exactly one `when` object and exactly two nested step lists:
  - `then`
  - `else`
- The `when` object must include:
  - `ref`
  - `op`
  - `value`
- Supported operators in this phase:
  - `equals`
  - `not_equals`
- `value` must be one scalar:
  - string
  - number
  - boolean
  - `null`
- `ref` may target only:
  - `steps.<node_id>.status`
  - `steps.<node_id>.output.<field_path>`
- Non-selected branch nodes remain visible in run detail and become `skipped` with `skip_reason_code=branch_not_taken`.

#### `end`

- Marks an explicit authored terminal outcome.
- Must contain `result`, one of:
  - `success`
  - `failure`
  - `noop`

### Manual Input Contract

Manual workflow input declarations live under `inputs`.

Each input item must include:

- `key`
- `type`
- `label`
- `required`

Supported `type` values in this phase:

- `text`
- `long_text`
- `number`
- `boolean`
- `select`
- `path`

Additional rules:

- `select` requires `options`, each with:
  - `label`
  - `value`
- `path` requires `mode`, one of:
  - `file`
  - `directory`
  - `either`
- `default` is optional for all input types.
- `key` must match `^[a-z][a-z0-9_]*$`.
- Input keys must be unique within one workflow.

Run snapshot freeze rules:

- Scalar inputs are copied into immutable snapshot input JSON.
- `path` inputs store:
  - the originally selected path
  - deterministic metadata
  - a copied snapshot payload under the run snapshot input store
- The snapshot store location must live under the run snapshot material root.
- For `directory` and `either` selections, snapshot copying must recurse only within the selected root.
- Snapshot copying for path inputs must exclude:
  - `.git/`
  - `.jj/`
  - `.origin/`
  - `node_modules/`
- Symlink targets outside the selected root are rejected deterministically.
- Execution and later inspection must consume the frozen snapshot copy, not the live original path.

Execution binding rules:

- All node input binding reads only from the immutable `run_snapshot`.
- The declared input `key` is the canonical identifier for both `{{inputs.<key>}}` prompt interpolation and `ORIGIN_INPUT_<KEY>` script environment binding.
- Execution may expose only captured snapshot values. The originally selected path remains audit and inspection metadata only.

### Resource Contract

Resources are declared at the workflow level and referenced by node `resource_id`.

Allowed resource reference shapes in this phase:

- Workflow-local resource:
  - `source: local`
  - `kind`
  - `path`
- Shared library resource:
  - `source: library`
  - `kind`
  - `item_id`

Allowed resource kinds for Phase `15` workflows:

- `script`
- `prompt_template`

Rules:

- Workflow-local resources live under `.origin/workflows/<workflow_id>/`.
- Workflow-local resource `path` values must remain relative to that workflow-local root.
- Shared library items remain live dependencies for future runs.
- Each run snapshot must freeze the exact material content used for every referenced shared and local resource.
- Future runs may use newer shared-library content.
- Past runs must continue to render and execute against the frozen snapshot materials they captured at start time.
- Existing library `query` items remain valid library assets from earlier phases, but they are not referencable from Phase `15` workflows and must fail validation with `resource_kind_unsupported`.

### Revision and Run Snapshot Contract

This phase introduces immutable workflow revisions, but it does not introduce workflow edit sessions or checkpoints.

Required persisted objects:

- `workflow_revision`
- `run_snapshot`
- `run_node`
- `run_attempt`
- `run_event`
- `session_link`

Explicitly deferred persisted objects:

- `workflow_edit_session`
- `workflow_checkpoint`
- `run_frame`

#### `workflow_revision`

- `workflow_revision` is the immutable stored record of one canonical workflow YAML text plus its content hash.
- A new `workflow_revision` must be created when:
  - the backend observes a valid `schema_version: 2` workflow file
  - its canonical text hash differs from the current stored head for that workflow ID
- Revision creation may happen lazily on validation, workflow detail load, or run start.
- There is no separate draft/publish mode in this phase.
- Manual file edits are the only workflow editing path in this phase.

#### `run_snapshot`

Each run snapshot must freeze:

- workflow ID
- workflow revision ID
- exact canonical workflow YAML text or content hash
- workflow-local resource materials used by the run
- shared-library resource materials used by the run
- immutable input payload
- input snapshot-store metadata for `path` inputs
- trigger metadata
- one exact parsed graph JSON derived from the same canonical workflow text as the stored revision fields

Additional rules:

- `workflow_revision_id`, `workflow_hash`, `workflow_text`, parsed graph JSON, and frozen resource materials for one `run_snapshot` must all derive from the same canonical workflow text observed at run start.
- Runtime must not reread the workflow file during run creation in a way that can mix an older validated graph with newer revision metadata.
- If the canonical workflow file changes after validation but before snapshot creation, run start must revalidate against one new coherent workflow view or fail before any runtime objects are persisted.

#### Top-Level Run Relationship

- The existing top-level `Run` object remains.
- Phase `07` and `08` top-level run states remain in force.
- The graph layer sits underneath the existing run lifecycle.
- Graph execution outcomes map back into the inherited top-level run states:
  - graph completes with no file changes -> `completed_no_change`
  - graph completes with file changes -> `ready_for_integration`
  - graph reaches authored `end.result=failure` or an unhandled node failure -> `failed`
  - cancellation continues to use inherited run cancellation behavior

This phase explicitly supersedes the Phase `07` implicit repair loop for graph workflows. `schema_version: 2` workflows do not perform any hidden post-run "repair in same session" behavior. All branching must be authored in the graph itself.

### Graph Runtime Contract

Each `run_node` must persist at minimum:

- `run_id`
- `snapshot_id`
- authored `node_id`
- `kind`
- `status`
- `skip_reason_code` nullable

Each `run_attempt` must persist at minimum:

- `run_node_id`
- `attempt_index`
- `status`
- linked execution session ID nullable

Each `run_event` must be ordered and linkable to:

- `run_id`
- optional `run_node_id`
- optional `run_attempt_id`

Required node status vocabulary:

- `pending`
- `ready`
- `running`
- `succeeded`
- `failed`
- `skipped`
- `canceled`

Required legal node transitions:

- `pending -> ready`
- `pending -> skipped`
- `pending -> canceled`
- `ready -> running`
- `ready -> skipped`
- `ready -> canceled`
- `running -> succeeded`
- `running -> failed`
- `running -> canceled`
- `running -> ready`

`running -> ready` is reserved for deterministic infrastructure retries only. It must not reopen a terminal node for hidden authored repair behavior.

Required attempt status vocabulary:

- `created`
- `running`
- `succeeded`
- `failed`
- `canceled`

Required legal attempt transitions:

- `created -> running`
- `running -> succeeded`
- `running -> failed`
- `running -> canceled`

Required skip reason codes in this phase:

- `branch_not_taken`
- `upstream_failed`

Additional runtime settlement rules:

- If cancellation is observed after a `run_node` or `run_attempt` has entered `running`, both records must transition to `canceled` before top-level run cancellation completes.
- If agent/script execution throws after a `run_attempt` is created, runtime must still persist terminal `failed` attempt/node state and ordered failure events before the top-level run settles.
- Any terminal workflow outcome must deterministically settle every remaining authored sibling or descendant that will not execute:
  - non-selected condition branches -> `skipped` with `skip_reason_code=branch_not_taken`
  - nodes abandoned because a failure terminal outcome or unhandled node failure stops progress -> `skipped` with `skip_reason_code=upstream_failed`
- Run completion, failure, or cancellation must not leave authored `run_node` or `run_attempt` records stranded in `pending` or `running` once execution has irreversibly stopped.

### Rerun Contract

Only one rerun product action is in scope:

- `Rerun workflow`

Rules:

- always creates a brand new run
- never mutates the source run
- always binds to the current live `workflow_revision`
- always binds to current live shared-library content
- may prefill inputs from a source run when launched from run detail
- still lets the user edit those inputs before starting

Explicitly out of scope:

- `Rerun from here`
- upstream node reuse
- downstream invalidation
- reconcile nodes

### Session-Link Contract

Workflow semantics must not depend on `parent_id`.

This phase introduces explicit runtime `session_link` records for workflow execution only.

Required stored fields:

- `session_id`
- `role`
- `visibility`
- `run_id` nullable
- `run_node_id` nullable
- `readonly`

Supported runtime roles in this phase:

- `execution_node`
- `run_followup`

Deferred authoring roles:

- `builder`
- `node_edit`

Role rules:

- `execution_node`
  - one session per agent-backed node attempt when a transcript exists
  - hidden from the global sidebar by default
  - visible from run-node detail
  - read-only by default
- `run_followup`
  - one persistent follow-up session per run by default
  - visible from the run page
  - hidden from the global sidebar by default

Additional rules:

- Power users may still open any linked session in the normal session route, but that route is never the workflow container.
- `Continue from here` from an execution transcript must never mutate the `execution_node` session itself.
- `Continue from here` must open or create the linked `run_followup` session seeded with run and node context.
- Applying workflow edits back into the canonical workflow file is out of scope for this phase.
- Session-link lookup APIs must remain scoped to the active project/workspace/run context. Raw `session_id` lookup must not expose hidden workflow session metadata outside the currently selected workspace/project.

### Navigation, History, and Notification Contract

Primary routes in scope:

- `/:dir/workflows`
- `/:dir/workflows/:workflowId`
- `/:dir/runs/:runId`
- `/:dir/library`
- `/:dir/history`
- `/:dir/session/:id` remains for normal chat usage

#### Workflow Detail

- default tab: `Design`
- required tabs:
  - `Design`
  - `Runs`
  - `Resources`
- this page is read-only in Phase `15`
- `Design` renders the current live graph and workflow metadata
- `Runs` links to run detail pages
- `Resources` renders local vs shared resource references and validation state

#### Run Detail

- default view: summary + full workflow snapshot graph
- summary must render captured `input_json` plus `input_store_json` snapshot metadata so frozen manual input values, including `path` snapshot provenance, remain inspectable after run start
- node detail opens in a side panel
- node panel query params are:
  - `node`
  - `panel`
  - `attempt`
- required panel values:
  - `summary`
  - `transcript`
  - `logs`
  - `artifacts`
  - `attempts`
- non-selected condition branches remain visible and dimmed with explicit skip reason

#### Global History

- keep:
  - `Runs`
  - `Operations`
  - `Drafts`
- `Open Run` actions must route to `/:dir/runs/:runId`
- no `Workflow Edits` tab is added in this phase

#### Notifications and Deep Links

- completed, failed, and canceled graph-workflow run notifications must deep-link to run detail
- hidden execution sessions must never be the primary notification target
- desktop deep-link plumbing must carry explicit route targets for:
  - workflow detail
  - run detail
- existing session deep links remain valid for generic chat usage only
- If hidden-session lookup is unavailable or fails, notification routing must fail closed for hidden workflow sessions rather than falling back to generic session-route notifications.

### Manual Run API Entry Contract

- `POST /workflow/run/validate` requires a JSON request body with:
  - `workflow_id`
- `POST /workflow/run/start` requires a JSON request body with:
  - `workflow_id`
  - `trigger_id` optional
  - `inputs` optional
- Server validators, OpenAPI descriptions, generated SDK request types, and generated SDK method signatures must preserve that required-body contract rather than weakening it to optional input.

### Lifecycle and Future-Phase Alignment Contract

- Workflow runs do not rewrite workflow definitions or library items by default.
- Workflow identity must survive filename changes.
- Manual runs bind to the current live `workflow_revision` at creation time and then execute through immutable `run_snapshot`.
- Future phases may extend `schema_version: 2` with:
  - additional trigger types
  - additional node kinds
  - workflow edit sessions/checkpoints
  - workflow edit history
  - library-item detail/editing
  - partial reruns
- Future phases `11` through `14` must attach behavior to:
  - `workflow_revision`
  - `run_snapshot`
  - `run`
  - `run_node`
  - `session_link`
  - workflow/run detail routes
- Future phases must not reintroduce a required "main run chat session" as the workflow execution container.

## Out of Scope

- `parallel`, `loop`, `validation`, and `draft_action` authored workflow nodes
- `Rerun from here`
- workflow edit sessions, checkpoints, and workflow edit history
- in-app graph editing
- `Build workflow with AI`
- `Edit graph`
- `Edit node with AI`
- builder/node-edit session roles
- applying run-followup changes back into the workflow file
- library item detail pages, `Used By`, resource history, and copy-on-write editing flows
- workflow duplication and workflow archival UX
- cron/signal execution behavior and scheduling runtime
- graph-node restart resume/replay semantics
- subworkflows
- runtime-generated graph topology
- cloud execution, multi-user collaboration, or non-desktop scope
- separate draft/publish workflow-definition modes
- arbitrary user-authored raw edge lists in YAML

## Baseline Assumptions

- The desktop-first, local-first, JJ-backed, DB-backed direction from Phases `01` through `13` remains in force.
- The repository is still pre-release, so this phase does not preserve backward-compatibility shims for `schema_version: 1` workflow execution.
- Existing top-level runtime entities remain authoritative:
  - `Run`
  - `Operation`
  - `Draft`
  - `IntegrationAttempt`
  - `DispatchAttempt`
  - `AuditEvent`
- Existing history tabs remain in place and are extended rather than replaced.
- Manual file edits are the only workflow editing path in this phase.
- `workflow_revision` creation may be lazy; the repo does not need a separate workflow-authoring watcher before this phase is implementable.
- Existing run cancellation, workspace cleanup, and integration-queue behavior from earlier phases remain in force unless this spec explicitly supersedes them.
- Graph workflows in this phase have no implicit post-run repair loop.
- `parent_id` may continue to exist for normal chat lineage, but it has no workflow-orchestration semantics in this phase.
- Only `execution_node` and `run_followup` `session_link` roles are active in this phase.

Specific earlier assumptions that this phase explicitly supersedes now:

- Phase `06` flat `instructions + resources` runnable workflow contract is replaced by `schema_version: 2` graph YAML.
- Phase `07` "one top-level mutable chat session per run" is replaced for graph workflows by `run` + `run_snapshot` + `session_link`.
- Phase `07` implicit "repair in same run session" behavior is removed for graph workflows.
- Phase `09` session deep links are no longer the primary navigation target for run execution; run detail becomes primary.

Specific earlier assumptions that remain inherited in this phase:

- Phase `08` queue/integration behavior still begins only after a graph run reaches the existing integration-ready boundary.
- Phase `09` `Runs`, `Operations`, and `Drafts` remain the global history tabs.
- Phase `10` outbound draft/dispatcher safety model remains unchanged.

## Required Keep Set

- Keep canonical workflow and library definitions on disk as user-visible YAML-backed files.
- Keep the current top-level runtime entities listed in `## Baseline Assumptions`.
- Keep the Draft/Dispatcher outbound safety model.
- Keep the app-wide session route and generic chat surface for non-workflow usage.
- Keep existing session/chat, diff, terminal, and history components where they can be reused inside workflow/run detail views.

## Deletion Candidates

Not applicable. This phase changes workflow foundations and execution surfaces but does not require repo pruning.

## Risk Register

- `schema_version: 1` cutover can break stale fixtures or internal sample workflows if the repo is not updated consistently in the same phase.
- The structured `condition.when` contract can become ambiguous if implementation quietly expands beyond the operators and reference grammar defined here.
- Path-input snapshot copying can become expensive or unsafe if ignore and symlink rules are implemented inconsistently.
- Hidden execution sessions can leak back into sidebar or notification behavior if the app reuses `parent_id` or root-session heuristics instead of `session_link`.
- If top-level run settlement and graph-node settlement drift apart, the app can show contradictory execution state across history, run detail, and integration surfaces.
