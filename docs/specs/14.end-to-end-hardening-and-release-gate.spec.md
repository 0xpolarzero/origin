# 14 Spec: End-to-End Hardening and Release Gate

## Goal

Establish a final implementation gate that validates the full feature matrix (manual runs, integration/recovery, drafts/outbound safety, triggers, debug flows, governance controls) with reproducible automated evidence before release.

## Why This Phase Exists

Even with strong per-phase tests, cross-phase interactions can still fail. This phase is the final confidence pass that proves the combined system behaves correctly under realistic and adversarial scenarios.

## In Scope

- Build full e2e matrix for:
  - happy path
  - regression path
  - negative/error path
- Validate restart/crash recovery interactions across integrated phases.
- Validate cross-surface UX coherence:
  - Runs/Operations/Drafts links
  - filter/default behavior
  - notification deep-link behavior
- Validate policy/security enforcement under end-to-end conditions.
- Add release checklist + go/no-go criteria.

## Out of Scope

- New product capabilities.
- New schema/runtime contracts unless critical defect requires fix.
- Large refactors not required for release blocking issues.

## Baseline Assumptions

- Phases `04` through `13` are implemented and logged.
- Local e2e harness is available and stable.
- Test fixtures can simulate trigger/event/recovery scenarios.
- Release-blocking severity rubric (critical/high/medium/low) is defined and shared before execution.
- Test flake policy is explicit (max retry budget, quarantine rules, and ownership).

## Agent Execution Plan

### Step 1: Build Unified E2E Scenario Matrix

- Create traceable matrix mapping each PRD invariant to one or more e2e tests.
- Mark test IDs and expected assertions.
- Commit matrix artifact at `/Users/polarzero/code/projects/origin/docs/specs/14.e2e-matrix.md` with columns:
  - `invariant_id`
  - `source_ref`
  - `test_id`
  - `scenario_class`
  - `result`

### Step 2: Implement Missing E2E Cases

- Add missing cases discovered in matrix coverage gap analysis.
- Prioritize negative/error and restart/recovery paths.

### Step 3: Run Full Validation Sweep

- Run package-level unit/typecheck suites.
- Run local e2e harness suites by scenario group.
- Capture deterministic artifacts for failures.

### Step 4: Release Gate Decision Pack

- Produce pass/fail summary.
- List blocking defects by severity and owning phase.
- Confirm go/no-go with explicit criteria.
- Include sign-off owner and timestamp.

## Acceptance Criteria

- `AC-01` Every critical PRD invariant has at least one automated e2e assertion.
  - Verification: matrix audit using committed artifact.
  - Pass condition: no critical invariants have empty `test_id`.

- `AC-02` Full lifecycle scenarios pass:
  - manual run -> integration -> history links
  - draft approve/send/reject/blocked flows
  - cron missed slot + retries
  - signal dedupe/no-backfill
  - debug reminder -> deep-link -> hard-stop path
  - non-origin outbound block
  - crash/restart recovery in integration
  - Verification: end-to-end suite.
  - Pass condition: all scenarios pass in local harness.

- `AC-03` Security/governance adversarial tests pass.
  - Verification: bypass/leakage canary suite.
  - Pass condition: no bypass and no secret leakage.

- `AC-04` No critical/high open defects remain after validation run.
  - Verification: consolidated defect report.
  - Pass condition: zero unresolved critical/high items.

- `AC-05` No skipped/quarantined tests remain in release-blocking suite.
  - Verification: test-run summaries and release gate report.
  - Pass condition: zero skipped/quarantined release-blocking scenarios.

- `AC-06` Release gate decision pack contains rubric-based outcome and ownership.
  - Verification: inspect decision artifact.
  - Pass condition: severity rubric, open-defect ownership, go/no-go owner, and timestamp are present.

- `AC-07` Release gate command suite is green.
  - Verification commands:
```bash
bun run --cwd packages/opencode typecheck
bun run --cwd packages/opencode test
bun run --cwd packages/app typecheck
bun run --cwd packages/app test:unit
bun run --cwd packages/app test:e2e:local -- e2e
bun run --cwd packages/desktop typecheck
bun ./packages/sdk/js/script/build.ts
git diff --exit-code
```
  - Pass condition: all commands pass and SDK generation leaves no unexpected drift.

## Risk Register

- E2E flakiness can mask real regressions or create false alarms.
- Recovery scenario simulation may miss platform-specific timing behavior.
- Final gate can balloon if prior phases under-test negative paths.
- Late-found defects may require cross-phase hotfix coordination.
