# 14 Spec: End-to-End Hardening and Release Gate

## Goal

Establish a final implementation gate that validates the full feature matrix (manual runs, JJ integration/recovery, generic drafts/dispatcher safety, triggers, debug flows, governance controls) with reproducible automated evidence before release.

## Why This Phase Exists

Even with strong per-phase tests, cross-phase interactions can still fail. This phase is the final confidence pass that proves the combined system behaves correctly under realistic and adversarial scenarios.

## In Scope

- Build full e2e matrix for:
  - happy path
  - regression path
  - negative/error path
- Validate restart/crash recovery interactions across integrated phases.
- Validate cross-surface UX coherence:
  - Runs/Operations/Drafts links
  - filter/default behavior
  - notification deep-link behavior
- Validate adapter/action schema behavior across Draft create/edit/approve/send flows.
- Validate policy/security enforcement under end-to-end conditions.
- Add release checklist + go/no-go criteria.

## Out of Scope

- New product capabilities.
- New schema/runtime contracts unless critical defect requires fix.
- Large refactors not required for release blocking issues.

## Baseline Assumptions

- Phases `04` through `13` are implemented and logged.
- Local e2e harness is available and stable.
- Test fixtures can simulate trigger/event/recovery scenarios.
- Release-blocking severity rubric for this phase:
  - `critical`: duplicate external side effect, policy bypass, secret leakage, destructive corruption, or unrecoverable data loss.
  - `high`: wrong terminal state, missing dispatch/audit provenance, broken approval/send safety gate, or failed recovery path in a release-blocking scenario.
  - `medium`: non-blocking regression with workaround or partial observability gap.
  - `low`: cosmetic, documentation, or non-blocking developer-experience issue.
- Test flake policy for this phase:
  - every release-blocking suite gets one immediate rerun after a failure
  - pass-on-rerun is treated as flake and still blocks release until fixed or explicitly removed from the release-blocking set by a spec delta
  - skipped or quarantined tests are not allowed in the release-blocking suite

### Step 2: Implement Missing E2E Cases

- Add missing cases discovered in matrix coverage gap analysis.
- Prioritize negative/error and restart/recovery paths.

### Step 3: Run Full Validation Sweep

- Run package-level unit/typecheck suites.
- Run local e2e harness suites by scenario group.
- Re-run flaky-prone and recovery-heavy suites per the declared flake policy.
- Capture deterministic artifacts for failures.

### Step 4: Release Gate Decision Pack

- Produce pass/fail summary.
- List blocking defects by severity and owning phase.
- Confirm go/no-go with explicit criteria.
- Include sign-off owner and timestamp.
- Commit decision artifact at `/Users/polarzero/code/projects/origin/docs/specs/14.release-gate-report.md`.

## Acceptance Criteria

- `AC-01` Every critical PRD invariant has at least one automated e2e assertion.
  - Verification: matrix audit using committed artifact.
  - Pass condition: no critical invariants have empty `test_id`.

- `AC-02` Every listed release-blocking lifecycle scenario has a named automated e2e and passes:
  - manual run -> JJ integration/recovery -> history links
  - one run emitting multiple Drafts with independent lifecycle outcomes
  - draft create/edit/approve/send/reject/blocked flows
  - `auto_approved` Draft send path
  - policy-eval failure -> `blocked` default-deny
  - sent Draft rejects further send attempts
  - blocked Draft -> fix/edit/re-approve -> new `dispatch_attempt`
  - adapter/action schema mismatch rejection
  - exactly-once `dispatch_attempt` handling
  - retry/recovery reuses the same dispatch idempotency key for one logical send
  - remote send accepted -> app crash before local finalize -> restart finalizes without duplicate send
  - system report Draft creation/approval/send uses outbound `dispatch_attempt`, not JJ `integration_attempt`
  - cron missed slot + retries
  - signal dedupe/no-backfill
  - debug reminder -> deep-link -> hard-stop path
  - non-origin outbound block
  - crash/restart recovery in JJ integration and outbound dispatch
  - Verification: end-to-end suite plus matrix audit for distinct `scenario_id` and `test_id` rows covering every listed scenario.
  - Pass condition: every listed scenario has a non-empty `scenario_id` + `test_id` pair in the committed matrix and passes in the local harness.

- `AC-03` Security/governance adversarial tests pass.

## Risk Register

- E2E flakiness can mask real regressions or create false alarms.
- Recovery scenario simulation may miss platform-specific timing behavior.
- Final gate can balloon if prior phases under-test negative paths.
- Late-found defects may require cross-phase hotfix coordination.
