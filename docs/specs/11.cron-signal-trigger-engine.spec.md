# 11 Spec: Cron and Signal Trigger Engine

## Goal

Implement automated workflow entry via cron and signal triggers with deterministic dedupe/retry/missed-run behavior and clear run recording semantics.

## Why This Phase Exists

After manual execution and outbound safety are stable, automation can be enabled safely. Trigger semantics must be deterministic to avoid duplicate work, silent drops, or confusing history.

## In Scope

- Cron trigger engine:
  - user timezone scheduling
  - missed-slot behavior (`skipped` + notify)
  - missed-slot expansion cap/coalescing policy for large offline gaps
  - deterministic DST behavior for spring-forward and fall-back boundaries
  - retry policy with retryable/non-retryable classes
- Signal trigger engine:
  - Origin workspace only
  - no backfill on enable
  - dedupe key enforcement
  - duplicate-event recording
  - fallback hash behavior when provider event ID is missing
  - explicit no-backfill boundary rule: process only events with `event_time > signal_enabled_at`
- Trigger-to-run creation pipeline:
  - new session per top-level run
  - run metadata includes trigger source
- Runs visibility for trigger outcomes:
  - skipped cron slots
  - duplicate signal event rows (`Ignored duplicate signal`)

## Out of Scope

- Reconciliation long-running reminder UX polish.
- Governance retention tier expansion.
- New integration adapter development.

## Baseline Assumptions

- Workflow definitions and manual execution path are stable.
- Deduplication persistence fields exist.
- Notification surfaces exist for skip/duplicate info.
- Trigger failure classes map to canonical machine-readable failure codes.
- Dedupe retention horizon (minimum 30 days) is greater than expected provider replay windows for this phase.
- Missed-slot coalescing policy in this phase:
  - generate at most 100 skipped rows per scheduler catch-up cycle
  - if more than 100 slots are missed, write one coalesced skipped summary row for the overflow
- DST policy in this phase:
  - spring-forward: nonexistent local-time slots are recorded as skipped (`dst_gap_skipped`)
  - fall-back: duplicated local-time slots are evaluated by unique UTC schedule boundaries (can yield two executions where applicable)
- Retry-class mapping in this phase:
  - retryable: `integration_transport_error`, `transient_runtime_error`, `integration_timeout`
  - non-retryable: `validation_error`, `schema_error`, `policy_blocked`, `workspace_policy_blocked`
- Persisted code contract for automated trigger failures:
  - retryable class exhausted after retries:
    - `run.failure_code=<class_code>`
    - `run.reason_code=retry_exhausted`
  - non-retryable class (no retry):
    - `run.failure_code=<class_code>`
    - `run.reason_code=non_retryable`
- Persisted code contract for non-executing trigger outcomes:
  - cron skipped slot: `run.failure_code` is null and `run.reason_code` is `cron_missed_slot` or `dst_gap_skipped`
  - duplicate signal event row: `reason_code=duplicate_event` and no run execution counter increment
- Fallback hash canonicalization contract:
  - canonical JSON with lexicographically sorted keys
  - UTF-8 encoding
  - adapter-owned field allowlist version tag included in hash input

## Signal Ingress Contract

- Signal transport in this phase is a backend-owned HTTP ingress route:
  - `POST /workflow/signals/:signal`
- Route scope and resolution rules:
  - `:signal` matches `workflow.trigger.signal` exactly
  - matching is workspace-local within the addressed Origin workspace only
  - one event may fan out to multiple runnable workflows in the same workspace that share the same `signal` value
  - standard workspaces must reject signal ingress for this phase with deterministic `workspace_policy_blocked` behavior
- Request body contract:
  - `event_time` is required and must be a Unix epoch millisecond integer
  - `provider_event_id` is optional and, when present, is the primary dedupe key component
  - `payload_json` is required and must be a JSON object
  - `source` is optional adapter/provider metadata object for audit/debug visibility only and must not affect dedupe identity except through explicit fallback-hash allowlist configuration
- Workspace and auth contract:
  - the request must target a concrete workspace via existing workspace scoping (`workspace` query/header)
  - this phase does not define a new public adapter auth model for signal ingress; the backend route is the canonical ingress boundary
- Enablement and no-backfill boundary:
  - `signal_enabled_at` is persisted per `(workspace_id, workflow_id)` when automated trigger registration first observes a runnable signal workflow for that workspace
  - updating workflow instructions/resources without changing `workflow_id` does not reset `signal_enabled_at`
  - removing a workflow and later reintroducing the same `workflow_id` creates a new enablement record with a new `signal_enabled_at`
  - only events with `event_time > signal_enabled_at` are eligible to create runs
- Dedupe identity contract:
  - dedupe scope is per `(workspace_id, workflow_id)`
  - if `provider_event_id` exists, dedupe key uses that ID
  - if `provider_event_id` is absent, dedupe key uses the fallback hash contract defined in `## Baseline Assumptions`
  - fallback hash input is `{ version, signal, event_time, payload_json, source }` after adapter-owned field allowlist pruning
- Unknown and invalid signal behavior:
  - unknown `:signal` values return success with `{ accepted: false, reason: "signal_unregistered" }` and create no run row
  - malformed requests return `400` and create no run row
  - events at or before enablement boundary return success with `{ accepted: false, reason: "before_enablement_boundary" }` and create no run row
- Duplicate behavior:
  - duplicates return success with `{ accepted: true, duplicate: true }`
  - duplicates must create one visible skipped run row per suppressed workflow with `reason_code=duplicate_event`
  - duplicate rows must not create sessions, must not increment execution counters, and must not increment run-attempt metrics

## Agent Execution Plan

### Step 1: Implement Cron Scheduling and Missed-Slot Logic

- Schedule by user timezone.
- On missed windows, record `skipped` run outcome and notify.
- Apply retry policy only for retryable failures.
- Persist terminal run `failure_code` and `reason_code` using the mapping contract in this phase.
- Apply missed-slot cap/coalescing when offline gap would create excessive skipped rows.
- Implement explicit DST boundary behavior (spring-forward/fall-back).
- Emit deterministic overflow summary when coalescing is applied.

### Step 2: Implement Signal Ingress and Dedupe

- Add the canonical route `POST /workflow/signals/:signal`.
- Enforce dedupe uniqueness key.
- If provider event ID absent, compute fallback hash from canonical adapter fields.
- Fallback hash canonicalization uses sorted-key JSON + UTF-8 + adapter field-set version tag.
- Duplicate event records no run execution and creates visible duplicate-event row.
- Reject or ignore events at/before enable timestamp (`event_time > signal_enabled_at`).

### Step 3: Trigger Metadata and Session Creation

- Ensure each automated run creates a new session.
- Persist trigger metadata and linkage IDs.

### Step 4: Notification/Visibility Integration

- Wire missed/duplicate outcomes into Runs surface and notifications.

### Step 5: Test Coverage

- cron scheduling and missed-slot
- retry classification
- signal dedupe and fallback hash
- no-backfill behavior

## Acceptance Criteria

- `AC-01` Missed cron slots create deterministic skipped outcomes.
  - Verification: simulate offline/missed interval.
  - Pass condition: deterministic skipped outcomes follow cap/coalescing policy; overflow emits one coalesced skipped summary row.

- `AC-02` DST boundary behavior is deterministic.
  - Verification: schedule tests around DST spring-forward/fall-back boundaries.
  - Pass condition: spring-forward and fall-back outcomes match documented DST policy and codes.

- `AC-03` Retry policy applies only to retryable classes and canonical failure-code mapping.
  - Verification: failure-class matrix tests.
  - Pass condition: retryability strictly matches declared class map, and persisted `run.failure_code`/`run.reason_code` values match the explicit mapping contract for retryable vs non-retryable classes.

- `AC-04` Signal dedupe suppresses duplicate execution.
  - Verification: submit duplicate signal payloads.
  - Pass condition: first event creates run, duplicates create `duplicate_event` rows only.

- `AC-05` Fallback hash dedupe works without provider event ID.
  - Verification: identical canonical payload with missing provider ID.
  - Pass condition: duplicates are detected reliably and hash stability is invariant to JSON key ordering.

- `AC-06` No backfill occurs on signal enable with strict boundary behavior.
  - Verification: enable signal on integration with historical backlog.
  - Pass condition: only events with `event_time > signal_enabled_at` create runs.

- `AC-07` Duplicate-event rows do not increment run/session execution counters.
  - Verification: duplicate ingestion tests with counter assertions.
  - Pass condition: counters remain unchanged for duplicate rows.

## Risk Register

- Timezone handling can drift around DST boundaries.
- Retry classification mistakes can overload integrations or drop recoverable runs.
- Weak canonicalization for fallback hashing can produce false negatives.
- Signal ingestion burst handling may stress dedupe persistence indexes.
