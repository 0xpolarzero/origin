# 07 Spec: Manual Run Execution

## Goal

Ship end-to-end manual workflow execution with session linkage and deterministic run lifecycle transitions, without yet introducing queued integration or automated triggers.

## Why This Phase Exists

This is the first user-visible execution slice. It validates the run model and session behavior before concurrency/recovery complexity is added.

## In Scope

- Manual trigger entrypoint for workflows.
- Create one new chat session per top-level run.
- Execute run in isolated run workspace.
- End-of-run validation flow:
  - success path
  - invalid-output repair loop in same session
- No-change vs change outcome classification.
- Persist run records with canonical lifecycle states.
- Persist integration-ready candidate metadata when file changes exist.
- Final operation linkage commit remains out of scope for this phase and is owned by Phase 08.
- Always perform run workspace cleanup on terminal outcomes.

## Out of Scope

- Integration queue/serialization and replay logic.
- Draft creation/dispatch.
- Cron and signal triggers.
- Hidden debug/reconciliation notification UX.

## Baseline Assumptions

- Runtime contracts (Phase 04) and JJ foundation (Phase 05) are available.
- Workflow/library validation (Phase 06) can mark runnable/non-runnable before execution.
- Existing session creation and prompt plumbing can be reused.
- Duplicate manual-trigger requests are rejected in this phase (no dedupe merge) with deterministic error and no second run/session creation.
- If cleanup fails, run terminal state remains stable and `cleanup_failed` is recorded for janitor retry.
- Partial-start failures follow rollback contract:
  - if session created but run workspace creation fails, mark run failed and keep session with explicit failure event
  - no orphan run workspace artifacts remain

## Agent Execution Plan

### Step 1: Add Manual Run Start Endpoint/Command

- Validate workflow is runnable.
- Create run record + linked session.
- Create isolated run workspace.

### Step 2: Execute Run in Session Context

- Run top-level instructions via base orchestration flow.
- Keep context in same run session thread.

### Step 3: Validation and Repair Loop

- Run final validation checks.
- If invalid, prompt same session for repair and retry validation.
- Keep retry attempts auditable in run events.
- Apply deterministic repair-loop exhaustion policy (attempt/time cap + terminal failure code).
- Repair-loop policy in this phase:
  - max 3 repair attempts OR 10 minutes elapsed from first repair prompt
  - terminal on exhaustion: `failed` with reason `repair_exhausted`

### Step 4: Persist Outcome and Cleanup

- If no file changes: `completed_no_change`, no operation row.
- If file changes: transition to `ready_for_integration` and persist integration-ready candidate metadata only.
- Cleanup run workspace for all terminal outcomes.

### Step 5: Test Coverage

- Manual run happy path.
- Validation-fail then repair path.
- No-change path.
- Cleanup on success/failure/cancel.
- Workspace isolation boundaries.
- Cancel during `running`/`validating`.

## Acceptance Criteria

- `AC-01` Manual run creation always creates a new linked session.
  - Verification: run start test asserts distinct `session_id` per run.
  - Pass condition: each manual run has a new session thread.

- `AC-02` Run lifecycle transitions are legal and deterministic.
  - Verification: unit/integration tests around explicit phase contract.
  - Pass condition: transition path is deterministic:
    - `queued -> running -> validating -> completed_no_change`
    - `queued -> running -> validating -> ready_for_integration`
    - `queued -> running|validating -> failed|canceled`

- `AC-03` Validation-fail path repairs in same session.
  - Verification: test fixture forcing invalid final state.
  - Pass condition: repair prompt/events occur in same `session_id`; exhaustion policy yields deterministic failure code when cap is hit.

- `AC-04` Duplicate manual-trigger requests are rejected deterministically.
  - Verification: issue duplicate manual trigger requests concurrently for same workflow/user action.
  - Pass condition: one run/session is created, subsequent request returns deterministic rejection.

- `AC-05` No-change runs do not create Operations entries.
  - Verification: run with no file mutations.
  - Pass condition: run ends `completed_no_change` and operations query returns zero new rows.

- `AC-06` Change-path runs persist integration-ready candidate metadata and do not finalize operations in this phase.
  - Verification: run with file mutations.
  - Pass condition: run ends `ready_for_integration` with required metadata and no finalized operation linkage row.

- `AC-07` Partial-start failure follows rollback/cleanup contract.
  - Verification: force failure after session creation and before run workspace creation completes.
  - Pass condition: run is terminal failed with explicit reason, session is accessible for diagnostics, no orphan workspace exists.

- `AC-08` All terminal outcomes trigger cleanup.
  - Verification: success/failure/cancel test matrix.
  - Pass condition: no orphaned run workspace remains.

- `AC-09` Manual run execution is isolated to run workspace boundaries.
  - Verification: run integration test with filesystem assertions.
  - Pass condition: mutations occur only inside run workspace until later integration phases.

- `AC-10` Validation commands pass.
  - Verification commands:
```bash
bun run --cwd packages/opencode typecheck
bun run --cwd packages/opencode test
bun run --cwd packages/app typecheck
bun run --cwd packages/app test:unit
bun run --cwd packages/app test:e2e:local -- e2e/session
```
  - Pass condition: all commands pass.

## Risk Register

- Repair loops can become noisy if validation conditions are too broad.
- Session linkage bugs can create orphan run/session records.
- Cleanup race conditions can leave stale run workspaces.
- Early execution path may expose latent prompt/tool regressions.
