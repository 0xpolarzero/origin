# 13 Spec: Security and Governance Hardening

## Goal

Harden runtime policy enforcement, secret handling, audit completeness, and outbound governance guarantees so behavior is provably safe within this non-sandboxed phase.

## Why This Phase Exists

Core behavior is implemented by earlier phases, but governance-grade guarantees need explicit hardening and adversarial tests to prevent policy bypass and sensitive-data leakage.

## In Scope

- Enforce strict outbound egress policy at runtime.
- Define and enforce adapter-owned dispatch registry and managed endpoint inventory matching rules for adapter-owned actions, schema versions, and targets.
- Expand secret redaction coverage to:
  - diagnostics
  - crash dumps
  - agent-visible tool output
  - exported reports
- Enforce audit completeness for policy/dispatch decisions.
- Enforce field allowlists for send-to-developers payloads.
- Enforce report target allowlist for system report Draft dispatches.
- Harden adapter/action payload validation against bypass attempts.
- Add explicit tests for bypass attempts and leakage canaries.

## Out of Scope

- Full sandboxing runtime.
- Long-term tamper-evident storage implementation.
- Enterprise/compliance export tooling.

## Baseline Assumptions

- Core draft/dispatch flow is already functional.
- Phase 10 acceptance criteria for draft gating are completed before this phase starts.
- Runtime has centralized dispatcher path for managed integrations with adapter-owned action schemas.
- Audit event baseline schema is available.

## Agent Execution Plan

### Step 1: Runtime Egress Enforcement

- Centralize managed outbound writes through dispatcher gate.
- Require valid `draft_id` + `dispatch_attempt_id` for managed endpoint writes.
- Reject direct agent/tool outbound writes to managed endpoints.
- Treat mismatched-but-structurally-valid IDs as policy bypass attempts and reject deterministically.
- Verify adapter/action/schema-version/target binding before transport execution.

### Step 2: Secret Redaction Expansion

- Add redaction utility coverage for logs/diagnostics/tool streams/reports.
- Add canary-based regression tests.

### Step 3: Audit Provenance Enforcement

- Require policy lineage fields for policy/dispatch events and full dispatch provenance for `dispatch.*` events.
- Reject incomplete audit writes for applicable event types.
- Fail closed on missing lineage for applicable events (no outbound dispatch proceeds).

### Step 4: Reporting Privacy Controls

- Enforce allowlisted payload fields.
- Enforce allowlisted report targets.
- Keep explicit consent and field preview mandatory.

### Step 5: Security Regression Suite

- Add adversarial tests for bypass attempts.
- Add canary secret leakage checks.

## Acceptance Criteria

- `AC-01` Managed outbound writes without valid draft context are rejected.
  - Verification: direct-call bypass tests.
  - Pass condition: writes rejected and audit events emitted.

- `AC-02` Forged context mismatches are rejected.
  - Verification: attempts with cross-workspace/cross-integration mismatched `draft_id` and `dispatch_attempt_id`.
  - Pass condition: request rejected with deterministic policy error and no outbound side effects.

- `AC-03` Adapter/action payload validation cannot be bypassed after Draft creation.
  - Verification: mutate stored Draft payloads or adapter/action/target bindings into incompatible combinations before dispatch, including tampering with derived fields such as `material_hash`, `preview_text`, or `block_reason_code`.
  - Pass condition: dispatcher blocks adapter mismatches with deterministic validation/policy error, recomputes or ignores derived fields, and produces zero outbound side effects.

- `AC-04` Secret canary never appears in diagnostics/crash/tool/report outputs.
  - Verification: canary injection tests across outputs.
  - Pass condition: no output contains unredacted canary value, including encoded/fragment variants.

- `AC-05` Policy lineage and dispatch provenance fields are mandatory for policy/dispatch events.
  - Verification: event-write validation tests.
  - Pass condition: incomplete events are rejected deterministically; `dispatch.*` events require `draft_id`, `dispatch_attempt_id`, `adapter_id`, `integration_id`, and `action_id` in addition to the required policy lineage set.

- `AC-06` Report payload and target allowlists are enforced.
  - Verification: attempt non-allowlisted fields/targets.
  - Pass condition: blocked with explicit reason code and no outbound write.

- `AC-07` Non-origin outbound attempts remain hard-blocked.
  - Verification: standard-workspace outbound attempt tests.
  - Pass condition: deterministic `workspace_policy_blocked` with user-visible remediation.

- `AC-08` Missing audit provenance fails closed and blocks dispatch.
  - Verification: induce missing lineage fields on policy/dispatch event write path.
  - Pass condition: outbound dispatch is blocked and zero side effects occur.

- `AC-09` Dispatch registry and managed endpoint inventory enforcement block out-of-inventory dispatches.
  - Verification: outbound attempts targeting undeclared adapter/action/schema-version/target combinations or targets/endpoints outside declared adapter-managed inventory.
  - Pass condition: request is rejected with deterministic policy reason and no write occurs.

## Risk Register

- Redaction false negatives can silently leak sensitive values.
- Over-redaction can reduce debuggability.
- Strict dispatcher enforcement can break legacy integrations if migration is incomplete.
- Adapter/action registry drift can create false allow/deny outcomes if versioning is weak.
- Allowlist management can become brittle without clear ownership.
