# 05 Spec: JJ Runtime Foundation

## Goal

Implement the app-managed JJ runtime layer (bootstrap, safe command wrappers, isolated run workspace lifecycle, and idempotent cleanup) without yet implementing workflow execution semantics.

## Why This Phase Exists

Later phases rely on JJ as checkpoint/runtime sidecar. If JJ interaction is not standardized first, command behavior, bootstrap mode, and cleanup semantics will drift and become unsafe.

## In Scope

- Add JJ runtime adapter with:
  - bundled runtime invocation path
  - canonical read-only command wrapper
  - canonical mutating command wrapper with telemetry hooks
- Implement bootstrap matrix behavior:
  - `.jj` exists
  - `.git` exists and `.jj` absent (colocated init)
  - no VCS metadata
- Add run workspace creation/forget/delete helpers.
- Implement idempotent cleanup behavior and telemetry outcomes.
- Implement poll/telemetry command policy:
  - read-only flows must use `--ignore-working-copy --at-op=@` where supported.
  - commands that cannot satisfy read-only safety policy are rejected with typed errors
- Add test seams for JJ command policy without introducing full run execution.

## Out of Scope

- Run lifecycle orchestration.
- Queueing/integration logic.
- Reconciliation loop.
- User-facing history/draft UI.
- Cron/signal behavior.

## Baseline Assumptions

- Bundled JJ binary/path mechanism is available in desktop runtime.
- Existing project/workspace abstractions can attach JJ adapter context.
- Local filesystem writes are allowed inside workspace path boundaries.
- One pinned JJ runtime version is used per platform build in this phase.
- Run workspace naming is deterministic and globally unique per workspace (`run_<run_id>`).
- JJ command policy is enforced at adapter boundary; touched call-sites do not invoke raw JJ shell commands directly.

## Agent Execution Plan

### Step 1: Introduce JJ Adapter Contracts

- Add `jj` adapter interface with explicit methods:
  - bootstrap
  - read-only command execution
  - mutating command execution
  - run workspace create/forget/delete
- Include structured result object with exit code, stderr classification, and telemetry tags.

### Step 2: Implement Bootstrap Matrix

- Implement deterministic bootstrap behavior for:
  - `.jj` present: no-op
  - `.git` only: initialize colocated JJ
  - neither: initialize JJ
- Ensure identity local-write rules from PRD are respected.

### Step 3: Add Read-Only Command Guard

- Centralize read-only wrapper:
  - supported read commands receive canonical flags
  - unsupported read commands are rejected with typed errors
- Reject unsupported read commands with typed errors.
- Reject or rewrite unsafe polling commands from caller paths.

### Step 4: Add Run Workspace Lifecycle Utilities

- Create run workspace helper with deterministic naming.
- Add cleanup helper:
  - forget workspace metadata first
  - delete path second
  - treat missing artifacts as success with warning telemetry
  - classify partial failures deterministically (warning vs error) for janitor retry

### Step 5: Test and Validation

- Add unit tests for bootstrap decision logic.
- Add unit/integration tests for command policy wrapper behavior.
- Add cleanup idempotency tests.

## Acceptance Criteria

- `AC-01` Bootstrap behavior is deterministic across `.jj`, `.git`, and empty directories.
  - Verification: matrix tests for all three states.
  - Pass condition: resulting JJ state matches spec, no destructive side effects occur, and existing `.git` refs/HEAD are preserved.

- `AC-02` Read-only polling wrapper enforces canonical no-snapshot policy.
  - Verification: tests execute read polling repeatedly.
  - Pass condition: no unintended JJ operation mutations are produced by polling paths.

- `AC-03` Unsupported read command invocations are rejected.
  - Verification: invoke read wrapper with commands lacking required safe semantics.
  - Pass condition: command is rejected with typed policy error (no fallback downgrade).

- `AC-04` Cleanup helper is idempotent.
  - Verification: run cleanup twice with already-forgotten/deleted artifacts.
  - Pass condition: second run succeeds with warning telemetry, not hard failure.

- `AC-05` Cleanup telemetry classification is deterministic.
  - Verification: force partial failure (forget succeeds, delete fails; and inverse).
  - Pass condition: telemetry classification and retry path match spec.

- `AC-06` Run workspace creation/cleanup APIs expose deterministic structured results.
  - Verification: unit tests validate result schema and telemetry fields.
  - Pass condition: no raw string-only error propagation.

- `AC-07` Touched package validation passes.
  - Verification commands:
```bash
bun run --cwd packages/opencode typecheck
bun run --cwd packages/opencode test
bun run --cwd packages/desktop typecheck
```
  - Pass condition: all commands pass.

## Risk Register

- JJ behavior differences across versions can invalidate command expectations.
- Wrapper overreach can block legitimate commands needed later.
- Cleanup handling may hide real failures if warning/error boundaries are too loose.
- Colocated bootstrap may behave differently on unusual repo layouts.
