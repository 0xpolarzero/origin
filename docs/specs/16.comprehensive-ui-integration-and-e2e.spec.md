# 16 Spec: Comprehensive UI Integration and E2E

## Goal

Deliver the release-grade UI phase that closes the remaining gap between Origin's approved graph-first product direction and the currently implemented thin inspection shell.

After this phase, the app must provide a first-class, fully integrated UI loop for:

- discovering workflows, library items, runs, history, and sessions
- authoring and editing workflows in-app
- validating and starting workflows with typed inputs
- rerunning whole workflows and selected failed portions
- inspecting runs, nodes, drafts, operations, and workflow edits
- continuing execution analysis through explicit session-role surfaces
- reviewing trigger/debug/notification/deep-link outcomes
- passing a strong local browser + integration-style + desktop-smoke verification matrix

## Why This Phase Exists

Phase `15` deliberately stopped at a foundation slice: canonical graph workflows, immutable run provenance, dedicated workflow/run detail routes, and read-only inspection surfaces. That foundation is implemented, but the app still behaves like a session-first desktop shell with thin workflow/library pages rather than a comprehensive workflow product.

The repo now has a clear mismatch:

- Approved direction already expects AI-first workflow authoring, graph editing, node editing, workflow edit history, library-item detail/history/copy-on-write, explicit builder/node-edit session roles, richer rerun semantics, and workflow-native notifications/deep links.
- The current app mostly exposes:
  - a validation list for workflows
  - a validation list for library items
  - a read-only workflow detail page
  - an inspection-oriented run detail page
  - strong history/draft/debug foundations, but still centered on thin global lists rather than a complete workflow operating loop
- The current local e2e harness is valuable but not sufficient as a release-grade UI gate:
  - it is browser-shell-only
  - many important flows still use mocked route interception
  - keyboard and accessibility coverage is shallow
  - there is no stricter integration profile that proves the new UI loop end-to-end

This phase exists to stop treating UI completion as a series of later follow-ups. It is the phase that brings every major missing or deferred UI surface needed for a coherent desktop product into one implementation plan.

### Missing Entirely

- Workflow creation entrypoints beyond "a YAML file exists on disk":
  - `Build workflow with AI`
  - explicit `New workflow`
  - builder session surface
- Workflow authoring/editing UI:
  - graph editing
  - node editing
  - scoped AI node/workflow edits
  - save/apply flow back to canonical files
- Manual run operating UI:
  - start-run modal/sheet
  - typed input collection
  - validation-before-start review
  - blocked-state remediation
  - rerun workflow
  - rerun from here
- Workflow edit history:
  - workflow detail `Edit History`
  - global `Workflow Edits`
  - checkpoints
  - edit-session drilldown
- Library item detail and editing:
  - `/:dir/library/:itemId`
  - content/config view
  - `Used By`
  - item history
  - edit shared item vs create local copy
  - delete-in-use blocking/remediation
- Dedicated runs discovery surface:
  - first-class runs list/index outside the current History tab-only experience
- Builder and `node_edit` session-role UX
- Durable notification/activity review UI beyond transient toast + badge state
- Strong accessibility/keyboard verification matrix for non-session surfaces

### Present But Thin / Incomplete

- Shell navigation exists, but it is still session/sidebar-centric and does not make workflows/library/runs/history feel like equal first-class destinations.
- Workflow list is a validation table with refresh only.
- Workflow detail is read-only with `Design`, `Runs`, and `Resources`, but no authoring, run, duplication, archival, or edit-history controls.
- Run detail is solid for inspection, but not for operation:
  - no start/rerun affordances
  - no input reuse/edit flow
  - no event-stream inspector
  - no run-node rerun controls
- Library list shows validation state but has no detail route or lifecycle management.
- History, operations, and drafts are the richest current workflow-adjacent surfaces, but:
  - operations drilldown is shallow
  - filtering is narrow
  - workflow edit history is absent
  - node attribution is limited
- Debug/reporting flows exist, but there is no stronger reconciliation inspector or durable activity surface.
- Deep links currently cover workflow/run/new-session/project targets, but not the broader UI matrix that this phase needs.
- Empty/loading/error/blocked states are inconsistent:
  - route suspense fallback is blank
  - session-empty is weak
  - richer blocked/remediation states are missing on workflow/library/run flows
- Keyboard and ARIA affordances are much stronger in session/prompt flows than in workflows/library/history/draft/editor flows.
- Local e2e coverage volume is decent, but it is not yet a release-grade UI gate because many high-value paths are mocked and browser-shell-only.

## Out of Scope

- Cloud-hosted, multi-user, or web-first workflow experiences.
- A separate draft/publish model for workflow definitions.
- Subworkflows.
- Runtime-generated graph topology.
- Interactive approval/input pause nodes.
- Full visual rule-builder authoring for conditions.
- Full sandboxing or new permission-profile systems beyond what is required to support the approved UI.
- Net-new production integration adapters unrelated to making the approved Draft/Dispatcher and report UI usable.
- Replacing the generic session route as the primary container for non-workflow chat usage.
- Backward-compatibility shims or migration/backfill work for pre-release data unless a touched UI flow cannot function without it.

## Baseline Assumptions

- Phase `15` foundation routes, runtime provenance objects, and graph contracts are the current implemented baseline, but this phase is allowed to extend them materially when required to make the approved UI real.
- Workflow and resource files on disk remain the canonical source of truth.
- Builder, node-edit, workflow-edit-history, library-detail, copy-on-write, trigger-authoring, and rerun-from-here surfaces are approved direction and are no longer treated as optional future UI for this phase.
- Settings may remain dialog-based if shell discoverability, command access, deep-linkability, and testability are preserved; the first-class routed-destination requirement applies specifically to workflows, library, runs, history, and sessions.
- Existing session/chat, diff, log, terminal, draft, and history components should be reused where they fit instead of replaced wholesale.
- Phase `16` may add or extend backend/runtime endpoints and SDK surfaces as needed so the UI is not implemented as mocked placeholder chrome.
- The repo remains pre-release. If current thin UI assumptions block a coherent comprehensive UI, prefer replacing them over preserving them.
- The local e2e harness may be extended with new documented flags or profiles, but the primary acceptance commands must still use `bun run --cwd packages/app test:e2e:local ...`.
