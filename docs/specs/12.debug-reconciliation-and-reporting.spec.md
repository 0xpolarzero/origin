# 12 Spec: Debug/Reconciliation Visibility and Reporting Flow

## Goal

Implement the hidden-by-default debug/reconciliation UX, long-running notifications, hard-stop behavior, deep-linking, and optional send-to-developers reporting flow.

## Why This Phase Exists

Reconciliation is intentionally abstracted for non-technical users, but the system still needs reliable observability and intervention controls when recovery runs long.

## In Scope

- Debug/reconciliation visibility controls:
  - hidden by default
  - reveal via `Show Debug Sessions`
  - compact hidden-count indicator
- Long-running notification behavior:
  - threshold trigger
  - repeat cadence
  - reminder actions (`Open debug session`, `Keep running`, `Stop and report`)
  - remaining-time text to hard stop
- Hard-stop behavior:
  - elapsed-time timeout
  - deterministic failure code
- `Open debug session` deep-link behavior:
  - auto-enable debug visibility in view
  - focus target run/session
- Stop/report flow:
  - field-level preview before consent
  - metadata default
  - allowlisted fields and destinations
  - explicit reporting channel model aligned with outbound gate (draft-backed system report draft)

## Out of Scope

- New reconciliation algorithm logic.
- Sandboxed execution.
- External crash-reporting infrastructure beyond existing integration paths.

## Baseline Assumptions

- Reconciliation runs are categorized as debug runs by runtime.
- Notification primitives and click-handling exist.
- Reporting path uses a draft-backed system channel aligned with outbound dispatch gate from Phase 10.
- Settings contract in this phase:
  - threshold default 15 minutes
  - reminder cadence default 10 minutes
  - hard stop default 45 minutes
  - invariant `hard_stop > threshold`

## Agent Execution Plan

### Step 1: Implement Debug Visibility Controls

- Add default hidden behavior for debug runs.
- Add `Show Debug Sessions` persistence and in-view toggle behavior.

### Step 2: Implement Long-Running Reminder Engine

- Trigger first reminder at configured threshold.
- Repeat at configured cadence.
- Include remaining-time text to hard stop.
- Cancel pending reminders immediately when run reaches terminal state.

### Step 3: Implement Hard-Stop Outcome

- Enforce timeout transition with deterministic failure code.
- Keep run/session linkage and audit trail intact.

### Step 4: Implement Deep-Link Behavior

- Ensure `Open debug session` action reveals and focuses target item.

### Step 5: Implement Stop/Report UX

- Add field-level preview.
- Require explicit consent.
- Apply allowlisted fields/destination rules.
- Enforce draft-backed reporting channel model with no gate bypass.

### Step 6: Test Coverage

- reminder timing and action behavior
- hard-stop state transition
- deep-link reveal/focus
- report-preview consent constraints

## Risk Register

- Timer scheduling drift can cause flaky reminder behavior.
- Hidden-by-default debug policy can reduce discoverability if indicators are weak.
- Report flow can accidentally leak fields if allowlist wiring is incomplete.
- Deep-link focus logic can break with list virtualization/pagination.
