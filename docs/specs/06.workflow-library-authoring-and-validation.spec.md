# 06 Spec: Workflow and Library Authoring with Validation

## Goal

Deliver authoring/parsing/validation for workflow and library resources so users can create runnable definitions with deterministic validation outcomes before execution features are enabled.

## Why This Phase Exists

Execution and trigger features depend on valid resource definitions. We need a stable authoring and validation contract first, including non-origin capability boundaries and collision rules.

## In Scope

- Add/extend YAML schemas for:
  - workflows
  - library resources (`query`, `script`, `prompt_template`)
- Implement parser/validator pipeline with explicit error codes.
- Enforce workspace capability constraints:
  - Origin vs standard workspace support rules
- Implement reference integrity validation:
  - missing resources
  - invalid kinds
  - broken links
- Implement knowledge-base import collision policy:
  - interactive: `Replace` / `Create copy` / `Cancel`
  - non-interactive (`cron`, `signal`): deterministic `Create copy` + notify
- Add UI validation surfacing hooks and runtime non-runnable handling for invalid definitions.
- UI validation contract in this phase:
  - Workflows and Library views show non-runnable state
  - error rows include `code`, `path`, and human-readable message
  - run entrypoints must reject non-runnable resources deterministically

## Out of Scope

- Running workflows.
- Queue/integration behavior.
- Draft dispatch.
- Cron/signal scheduling runtime.

## Baseline Assumptions

- `.origin/workflows/*.yaml` and `.origin/library/*.yaml` remain canonical file storage.
- Existing config/markdown helpers can be reused for parsing and diagnostics formatting.
- Workspace type is already available and immutable in runtime state.
- This phase owns a canonical validation error-code catalog consumed by parser, runtime, and UI.
- Notification pipeline for automated collision outcomes is available to this phase.
- Path normalization/case-sensitivity policy is explicitly applied for collision checks.
- Schema-version compatibility behavior for older YAML is explicit (supported/fail with code).

## Agent Execution Plan

### Step 1: Define Workflow/Library Schemas

- Add versioned schema definitions and parsers.
- Validate required fields and enum values.
- Attach machine-readable error codes for every validation failure class.

### Step 2: Implement Validation Service

- Build validation entrypoint that returns:
  - normalized resource object
  - error list with code/path/message
  - runnable flag
- Enforce non-origin capability rules.

### Step 3: Implement Reference and Collision Logic

- Validate references between workflow and library assets.
- Add deterministic collision handling strategy for knowledge-base imports.

### Step 4: Integrate Validation Surfacing

- Wire errors to UI/runtime views as explicit non-runnable states.
- Ensure invalid definitions never execute.
- Ensure non-runnable state is visible in Workflows and Library surfaces with deterministic code/path rendering.

### Step 5: Test Coverage

- Parser and schema tests.
- Reference integrity tests.
- Workspace capability boundary tests.
- Collision policy tests for interactive and automated contexts.

## Acceptance Criteria

- `AC-01` Valid definitions parse and normalize successfully.
  - Verification: unit tests for each schema kind with representative fixtures.
  - Pass condition: normalized output matches expected shape.

- `AC-02` Invalid definitions are non-runnable with deterministic error codes.
  - Verification: invalid fixture matrix.
  - Pass condition: each invalid case emits stable code/path and runnable=false.

- `AC-03` Non-origin workspace capability constraints are enforced.
  - Verification: tests attempt unsupported resource usage in standard workspace.
  - Pass condition: deterministic validation error and non-runnable result.

- `AC-04` Reference integrity validation catches missing/broken links.
  - Verification: fixtures with missing resource and wrong kind references.
  - Pass condition: errors include reference path and reason code.

- `AC-05` Knowledge-base collision handling is deterministic.
  - Verification:
    - interactive flow test for `Replace`/`Create copy`/`Cancel`
    - automated flow test for deterministic `Create copy`
  - Pass condition: resulting file state and notifications match policy and include action/result metadata.

- `AC-06` UI surfaces invalid definitions deterministically.
  - Verification: component/e2e tests for invalid workflow/library definitions.
  - Pass condition: non-runnable state and `code`/`path`/message are shown consistently.

- `AC-07` Runtime rejects execution attempts for invalid definitions.
  - Verification: invoke validation gate used by execution entrypoints with invalid workflow reference.
  - Pass condition: deterministic error code returned and no run is started.

- `AC-08` Validation command suite passes.
  - Verification commands:
```bash
bun run --cwd packages/opencode typecheck
bun run --cwd packages/opencode test
bun run --cwd packages/app typecheck
bun run --cwd packages/app test:unit
bun run --cwd packages/app test:e2e:local -- e2e/workflows
```
  - Pass condition: all commands pass.

## Risk Register

- Schema evolution may require migration helpers if early definitions are incomplete.
- Validation strictness can block legitimate legacy files unexpectedly.
- Error-code instability can make downstream e2e tests brittle.
- Collision logic can become inconsistent across platforms if path normalization is weak.
